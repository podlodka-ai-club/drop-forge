## Context

Проект уже документирует базовые команды проверки в `README.md`: `go fmt ./...` и `go test ./...`. Процент покрытия сейчас не отображается на GitHub и не закреплен в спецификациях. Запрос DRO-24 требует видимый процент покрытия в README или другом заметном месте.

Кодовая база на Go, поэтому расчет покрытия можно выполнить штатными инструментами Go: `go test` с `-coverprofile` и `go tool cover -func`. Это не требует новых runtime-параметров, секретов или внешних сервисов.

## Goals / Non-Goals

**Goals:**

- Показать актуальный на момент изменения процент тестового покрытия в корневом `README.md`.
- Добавить воспроизводимую команду для локального пересчета покрытия.
- Сохранить текущий минимальный workflow проверки проекта.
- Не привязывать решение к внешнему SaaS до появления требования на автоматическую публикацию покрытия.

**Non-Goals:**

- Автоматически обновлять README после каждого push.
- Вводить обязательный порог покрытия для CI.
- Добавлять сторонние сервисы бейджей, если процент можно показать без них.
- Менять бизнес-логику CLI, `TaskManager`, `AgentExecutor` или runtime-конфигурацию.

## Decisions

1. **Источник процента покрытия: стандартный Go coverage.**
   - Решение: считать покрытие командой вида `go test ./... -coverprofile=coverage.out` и извлекать общий процент через `go tool cover -func=coverage.out`.
   - Причина: это штатный инструмент Go, он уже доступен вместе с проектным toolchain и не добавляет зависимостей.
   - Альтернатива: подключить внешний coverage-сервис. Отклонено как преждевременное усложнение для задачи, где нужен видимый процент на GitHub.

2. **Место публикации: корневой README.**
   - Решение: добавить заметный элемент в верхнюю часть `README.md` и раздел в проверках, объясняющий пересчет.
   - Причина: README уже является главным входом в проект на GitHub и покрыт существующей capability `project-readme`.
   - Альтернатива: отдельный документ в `docs`. Отклонено, потому что показатель должен быть виден сразу на странице проекта.

3. **Формат отображения: простой процент с понятным источником.**
   - Решение: использовать текстовый индикатор или badge-like строку, значение которой явно связано с командой пересчета.
   - Причина: статический README не должен создавать впечатление автоматического обновления, если автоматизации нет.
   - Альтернатива: динамический badge через GitHub Actions или Shields endpoint. Откладывается до отдельной задачи на CI-автоматизацию.

4. **Артефакты покрытия не хранятся в репозитории по умолчанию.**
   - Решение: временный `coverage.out` считается локальным артефактом и не становится частью публичной документации.
   - Причина: файл профиля покрытия шумный, зависит от текущего прогона и не нужен для отображения процента.

## Risks / Trade-offs

- Статический процент может устареть после изменения тестов или кода -> в README нужно указать команду пересчета, а в задачах реализации добавить проверку, что значение обновлено после локального прогона.
- Без CI нет автоматической защиты от снижения покрытия -> это осознанно вне scope; можно добавить отдельной задачей, когда появится требование на quality gate.
- `go test ./... -coverprofile` может учитывать пакеты без тестов иначе, чем ожидает команда -> спецификация должна фиксировать именно штатный Go-расчет для всех пакетов, чтобы не было скрытой кастомной методики.
