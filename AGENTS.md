# AGENTS.md

## Назначение проекта

Этот репозиторий развивает оркестратор для кодинг-агентов.
Подход итеративный: начинаем с простого работающего решения, затем усложняем архитектуру только при реальной необходимости.

## Язык и стек

- Основной язык проекта: Go.
- Новые сервисы, утилиты и CLI по умолчанию писать на Go.
- Если нужен внешний инструмент, сначала обосновать, почему стандартной библиотеки Go недостаточно.

## Коммуникация

- Внутри проекта предпочтителен русский язык.
- Обсуждение задач, планы изменений и пояснения к реализации писать на русском.
- Спеки в `openspec` стараться писать на русском, если нет явной причины использовать английский.
- В коде допустим английский для идентификаторов, имен пакетов, API и технических терминов.

## Правила разработки

- Держать решения простыми и расширяемыми.
- Не добавлять абстракции заранее без подтвержденной необходимости.
- Перед изменением поведения сначала понять текущий поток данных и точки расширения.
- Следовать идиомам Go: маленькие пакеты, явные зависимости, простые интерфейсы.
- После изменений обязательно прогонять форматирование и тесты.

## Обязательные команды перед завершением задачи

Минимальный обязательный набор:

```bash
go fmt ./...
go test ./...
```

Если появятся дополнительные проверки, их нужно документировать и запускать вместе с тестами.

## Конфигурация

- Все runtime-параметры выносить в `.env`.
- В репозитории обязательно поддерживать актуальный `.env.example` без секретов.
- В `.env.example` хранить только ключи переменных без значений по умолчанию.
- При добавлении новой переменной обновлять и `.env.example`, и код загрузки конфигурации.
- Нельзя хардкодить токены, ключи, URL окружений и прочие runtime-настройки.

Базовые ожидаемые файлы:

- `.env` — локальные значения для разработки.
- `.env.example` — шаблон переменных без секретов.

## Структура и рост проекта

- На раннем этапе предпочитать простую структуру каталогов.
- Выделять пакеты и модули только когда появляется явная ответственность или переиспользование.
- Для новых функций сначала собирать минимально рабочий сценарий, затем расширять наблюдаемость, отказоустойчивость и параллелизм.

## Архитектурная рамка

- Внутренними акторами проекта считаем только `TaskManager`, `CoreOrch`, `AgentExecutor`, `GitManager`, `Logger`.
- `TaskManager` изолирует проект от конкретного task backend. Сегодня это может быть `Linear`, но код проекта должен зависеть от своего контракта, а не от SDK конкретной системы.
- `AgentExecutor` изолирует проект от конкретного coding-agent runtime. Сегодня это `CodexCLI`, но orchestration-слой не должен быть жестко привязан к нему.

Целевой поток для proposal-stage:

1. `CoreOrch` регулярно запрашивает у `TaskManager` задачи, готовые к proposal-обработке.
2. `TaskManager` читает задачи из внешнего tracker-а и возвращает внутреннюю модель задач.
3. `CoreOrch` передает задачу в `AgentExecutor`.
4. `AgentExecutor` через `GitManager` поднимает isolated workspace, запускает агент и получает результат.
5. Во время выполнения `AgentExecutor` пишет шаги и stdout/stderr событий в `Logger`.
6. После изменений `GitManager` оформляет branch, commit, push и PR.
7. `AgentExecutor` возвращает `CoreOrch` результат выполнения, включая `PR URL`.
8. `CoreOrch` просит `TaskManager` обновить статус задачи и прикрепить ссылку на PR.
9. Если человек отклоняет proposal, задача возвращается в `Ready to Propose` и цикл повторяется.
10. Если человек принимает proposal, задача переводится в `Ready to Code`.

Границы ответственности:

- `CoreOrch` координирует сценарий, но не должен содержать детали `git`, `gh`, `codex` или API task tracker-а.
- `TaskManager` отвечает за поиск задач, смену статусов, комментарии и привязку артефактов задачи.
- `AgentExecutor` отвечает за lifecycle агентного запуска: подготовка input, запуск, сбор результата, публикация логов, возврат статуса.
- `GitManager` отвечает за операции с repository/workspace: `clone`, ветки, commit, push, PR.
- `Logger` отвечает за единый формат структурных событий и за доставку логов в sink-и.

## Маппинг На Текущий Код

- При создании, выделении или существенном изменении сервисов агент обязан обновлять эту секцию, чтобы статус реализации и маппинг на код оставались актуальными.

- `Logger` уже реализован в `internal/steplog`. Это текущий готовый сервис с явным контрактом JSON Lines.
- `AgentExecutor` частично реализован в `internal/proposalrunner`: именно этот модуль сейчас запускает `codex`, собирает последний ответ и управляет шагами workflow.
- `GitManager` пока не выделен в отдельный пакет, но его ответственность уже фактически присутствует внутри `internal/proposalrunner` через команды `git clone`, `checkout -b`, `add`, `commit`, `push` и `gh pr create`.
- `CoreOrch` пока существует в упрощенном виде как CLI entrypoint `cmd/orchv3/main.go`, который принимает вход, загружает конфиг и инициирует единичный orchestration run через `proposalrunner.Run`.
- `TaskManager` пока не реализован как сервис в репозитории. Сейчас его роль находится вне системы и выполняется вручную или через внешний tracker.
- `internal/commandrunner` — это не отдельный доменный актор, а технический адаптер для запуска внешних команд, который уже переиспользуется `AgentExecutor`/будущим `GitManager`.

Текущее архитектурное чтение репозитория:

- сегодня проект уже покрывает вертикальный slice `CoreOrch -> AgentExecutor -> GitManager -> Logger` для одного proposal-run без постоянного polling loop;
- следующий естественный шаг роста — отделить из `internal/proposalrunner` самостоятельные `GitManager` и `AgentExecutor`, а затем добавить `TaskManager` и долгоживущий `CoreOrch`;
- до появления реальной потребности не выделять новые сервисы сверх этих пяти ролей.

## Работа со спеками

- Спеки хранить в `openspec`.
- Существенные изменения поведения сначала описывать в спеках и задачах.
- В спеках фиксировать цель, ограничения, сценарии и критерии приемки.

## Тестирование

- Любое изменение логики должно сопровождаться тестами или понятным объяснением, почему тест здесь пока неуместен.
- Предпочтение: table-driven tests для бизнес-логики и понятные интеграционные тесты для сценариев оркестрации.
- Если тесты не запускаются из-за инфраструктурного ограничения, это нужно явно указать в итоговом отчете.

## Практические договоренности

- Логику конфигурации держать централизованно.
- Ошибки оборачивать с контекстом.
- Публичное поведение CLI и API документировать по мере появления.
- Секреты, временные файлы и локальные артефакты не коммитить.
