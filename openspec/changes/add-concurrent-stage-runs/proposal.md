## Why

Сейчас общий orchestration pass обрабатывает задачи proposal, apply и archive последовательно, поэтому одна долгая агентская операция блокирует запуск следующей готовой задачи. Нужно дать оркестратору возможность одновременно вести как минимум две независимые задачи, не меняя внешний workflow Linear и OpenSpec.

## What Changes

- Добавить параллельный запуск задач proposal, apply и archive в отдельных горутинах внутри orchestration pass.
- Ограничить число одновременно выполняемых задач контролируемым лимитом, чтобы по умолчанию можно было вести две задачи одновременно.
- Сохранить текущие state transitions: задача должна переходить в in-progress до запуска соответствующего executor'а и в review-state только после успешного завершения.
- Собирать и возвращать ошибки отдельных задач так, чтобы отказ одной задачи не отменял уже запущенные независимые задачи.
- Добавить структурные логи старта, завершения и ошибки для конкурентных task runs.
- Покрыть конкурентное поведение тестами без реальных Linear, GitHub, git, Codex или OpenSpec вызовов.

## Capabilities

### New Capabilities

- Нет.

### Modified Capabilities

- `proposal-orchestration`: общий orchestration runtime должен запускать готовые proposal/apply/archive задачи конкурентно с ограничением параллелизма вместо строгой последовательной обработки всего списка.

## Impact

- `internal/coreorch`: планирование task runs, лимит параллелизма, сбор ошибок, логи и unit-тесты для конкурентного выполнения.
- `cmd/orchv3`: wiring runtime-настройки параллелизма, если она потребуется как конфигурационный параметр.
- `internal/config` и `.env.example`: возможное добавление переменной лимита параллелизма без значения по умолчанию в `.env.example`.
- `openspec/specs/proposal-orchestration/spec.md`: уточнение требований к обработке нескольких proposal/apply/archive задач в одном pass.
- `architecture.md`: обновить только если реализация меняет границы ответственности компонентов orchestration runtime.
