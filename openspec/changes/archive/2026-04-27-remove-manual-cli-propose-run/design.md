## Context

Сейчас `cmd/orchv3/main.go` поддерживает два публичных режима: прямой single-run запуск `proposalrunner.Run` по описанию задачи из args/stdin и явную команду `orchestrate-proposals`, которая выполняет один проход `CoreOrch`. Архитектурная цель проекта уже описывает регулярный цикл `CoreOrch -> TaskManager -> AgentExecutor`, но в коде пока нет долгоживущего polling loop.

DRO-32 меняет точку входа: CLI больше не должен принимать ручное описание задачи для первой тестовой команды. Основной запуск приложения должен постоянно мониторить Linear-состояние `Ready to propose` и запускать существующий proposal orchestration pass.

## Goals / Non-Goals

**Goals:**

- Удалить публичный CLI-путь прямого запуска proposal runner по args/stdin.
- Сделать основной CLI-запуск долгоживущим polling loop поверх существующего `RunProposalsOnce`.
- Настраивать интервал polling через `.env` и валидировать его при загрузке конфигурации.
- Сохранить testability: цикл должен принимать заменяемый timer/sleeper или ограничитель итераций в тестах, чтобы unit-тесты не зависали.
- Обновить `architecture.md`, потому что меняется orchestration flow на уровне entrypoint и runtime lifecycle.

**Non-Goals:**

- Не менять внутренний workflow `proposalrunner`: clone, Codex, git, PR и comment остаются как есть.
- Не добавлять параллельную обработку задач; текущая последовательная обработка сохраняется.
- Не реализовывать мониторинг `Ready to code` или `Ready to archive` в рамках DRO-32.
- Не добавлять новый внешний scheduler или daemon framework.

## Decisions

1. CLI без подкоманд запускает continuous proposal monitor.

   - Решение: `runWithDeps` после загрузки конфигурации всегда wiring-ит `TaskManager`, `proposalrunner` и `coreorch`, затем запускает бесконечный loop.
   - Альтернатива: оставить команду `orchestrate-proposals` и добавить новую команду `monitor-proposals`. Это сохраняет ручной запуск как норму и хуже соответствует задаче убрать CLI-запуск первой тестовой команды.
   - Последствие: args/stdin больше не являются task description input. Непустые CLI-аргументы должны приводить к понятной ошибке usage, чтобы старый ручной режим не сработал молча.

2. Loop вызывает существующий `RunProposalsOnce`.

   - Решение: добавить небольшой слой `RunProposalsLoop(ctx, interval)` в `coreorch` или рядом с CLI wiring, который логирует старт итерации, вызывает `RunProposalsOnce`, логирует ошибку и продолжает после паузы.
   - Альтернатива: встроить `for` прямо в `cmd/orchv3/main.go`. Это быстрее, но усложнит тестирование и смешает lifecycle loop с CLI wiring.
   - Последствие: `RunProposalsOnce` остается отдельным методом для unit-тестов и одной итерации loop, но не выставляется как ручной CLI-режим.

3. Ошибка одной итерации не завершает процесс.

   - Решение: если загрузка конфигурации или wiring зависимостей не удались, CLI завершает процесс с кодом 1; если ошибка произошла внутри polling iteration, loop пишет structured error и продолжает после интервала.
   - Альтернатива: завершать процесс при первой ошибке обработки задачи. Это проще, но ломает идею долгоживущего мониторинга и требует внешнего supervisor-а для временных сбоев Linear/GitHub/Codex.
   - Последствие: при устойчивой ошибке процесс будет регулярно повторять попытки и писать ошибки. Частота ограничивается polling interval.

4. Интервал polling задается runtime-конфигурацией.

   - Решение: добавить `PROPOSAL_POLL_INTERVAL` как `time.Duration`-строку (`30s`, `1m`) с безопасным default в коде и пустым ключом в `.env.example`.
   - Альтернатива: использовать фиксированный hardcoded interval. Это нарушает проектное правило выносить runtime-параметры в `.env`.
   - Последствие: конфигурация должна валидировать положительное значение и покрываться тестами `config`.

## Risks / Trade-offs

- [Риск] Бесконечный loop может зависнуть в тестах или локальном запуске без настроенного окружения -> [Mitigation] вынести loop в тестируемый слой с context cancellation и fake sleeper/ticker; конфигурационные ошибки должны происходить до входа в loop.
- [Риск] Старые пользователи CLI могут продолжить передавать описание задачи аргументами -> [Mitigation] явно возвращать ошибку о том, что ручной proposal запуск удален, вместо трактовки args как задачи.
- [Риск] Повторяющиеся ошибки внешних API будут засорять логи -> [Mitigation] логировать одну structured error на итерацию и управлять частотой через `PROPOSAL_POLL_INTERVAL`.
- [Trade-off] Не добавляем отдельный scheduler-сервис или backoff-механику сейчас -> это проще и достаточно для первой версии постоянного мониторинга; backoff можно добавить позже при реальной необходимости.

## Migration Plan

1. Обновить конфигурацию и `.env.example`, добавив `PROPOSAL_POLL_INTERVAL`.
2. Добавить loop API поверх `RunProposalsOnce` с context cancellation и тестируемым ожиданием.
3. Перевести CLI default path на loop и удалить прямое чтение task description из args/stdin.
4. Обновить тесты CLI, config, orchestration loop и `architecture.md`.
5. Запустить `go fmt ./...` и `go test ./...`.

Rollback: вернуть CLI wiring к одноразовой команде `orchestrate-proposals` и прямому proposal runner path из предыдущей версии, если постоянный мониторинг окажется непригодным для окружения.

## Open Questions
