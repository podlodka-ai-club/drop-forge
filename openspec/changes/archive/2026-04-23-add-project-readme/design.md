## Context

Проект представляет собой Go CLI `orchv3`, который в текущем виде умеет принимать описание задачи, запускать proposal workflow и возвращать URL созданного pull request. Информация о запуске и внешних prerequisites уже частично есть в `docs/proposal-runner.md`, но у репозитория нет корневого README, который давал бы быстрый вход в проект.

Для будущего README уже есть несколько источников правды:

- `cmd/orchv3/main.go` показывает фактические способы запуска: через CLI-аргументы и через `stdin`;
- `internal/config/config.go` определяет обязательные и опциональные runtime-настройки;
- `.env.example` содержит полный список поддерживаемых переменных окружения;
- `go.mod` фиксирует Go-модуль и внешние Go-зависимости;
- `docs/proposal-runner.md` содержит детальное описание proposal runner и его prerequisites.

Изменение относится не к runtime-поведению, а к репозиторной документации. Поэтому важно не дублировать детали без нужды и не придумывать сценарии, которых нет в коде.

## Goals / Non-Goals

**Goals:**

- Добавить в корень репозитория `README.md` как основную точку входа для разработчика.
- Кратко и точно описать назначение `orchv3` и текущий пользовательский сценарий.
- Зафиксировать минимальные шаги запуска с примерами команд.
- Явно перечислить внешние зависимости и требования к окружению.
- Показать, где искать конфигурацию и какие команды использовать для базовой проверки проекта.
- Сослаться на `docs/proposal-runner.md` для более детального описания workflow, не перенося туда весь текст целиком.

**Non-Goals:**

- Не менять код приложения, конфигурацию или runtime-поведение CLI.
- Не дублировать в README всю внутреннюю реализацию proposal runner.
- Не документировать несуществующие режимы запуска, HTTP API или деплой, которых сейчас нет в коде.
- Не добавлять новую структуру документации за пределами корневого README, если существующего `docs/proposal-runner.md` достаточно.

## Decisions

### README как entrypoint, `docs/proposal-runner.md` как детальная справка

В корневом `README.md` должна быть сжатая, прикладная информация: что это за проект, как его запустить, какие зависимости нужны, как настроить `.env`, где лежит подробная документация. Детали proposal workflow, которые уже хорошо описаны в `docs/proposal-runner.md`, лучше оставить там и дать ссылку из README.

Альтернатива: перенести весь `docs/proposal-runner.md` в README. Это сделает корневой документ перегруженным и усложнит поддержку.

### Содержимое README строить только от текущего кода и конфигурации

README должен описывать только подтвержденные сценарии:

- запуск бинаря с описанием задачи в аргументе;
- запуск с передачей задачи через `stdin`;
- печать PR URL в `stdout`;
- шаговые логи в `stderr`;
- зависимость от `.env`, `git`, `codex`, `gh`, Go и GitHub authentication.

Альтернатива: писать README "на вырост" с гипотетическими режимами. Это увеличит риск устаревшей документации.

### В README явно выделить разделы для быстрого старта и для разработки

Минимальный набор разделов:

- назначение проекта;
- зависимости;
- настройка окружения;
- запуск;
- структура проекта или ключевые директории;
- команды проверки (`go fmt ./...`, `go test ./...`);
- ссылки на связанные документы.

Такой набор покрывает запрос пользователя и одновременно помогает новому участнику быстро войти в репозиторий.

Альтернатива: ограничиться только запуском и зависимостями. Это формально закроет запрос, но не решит проблему базовой навигации по проекту.

## Risks / Trade-offs

- [README начнет расходиться с кодом] -> опираться на `main.go`, `config.go`, `.env.example` и `go.mod` как на источники правды и не описывать неподтвержденные возможности.
- [Дублирование информации с `docs/proposal-runner.md`] -> держать в README только краткое описание и ссылаться на существующий документ за деталями.
- [README станет слишком абстрактным] -> добавить конкретные команды запуска и явный список внешних CLI-зависимостей.
- [README быстро устареет при расширении CLI] -> включить в документ раздел с текущими ограничениями и не обещать будущие сценарии.

## Migration Plan

1. Создать `README.md` в корне репозитория.
2. Перенести в него проверенную информацию о назначении, запуске, зависимостях и конфигурации.
3. Добавить ссылки на `.env.example`, `docs/proposal-runner.md` и ключевые директории проекта.
4. Проверить, что README не противоречит текущему коду и существующей документации.

Откат тривиален: удалить или переписать `README.md`, так как изменение не затрагивает runtime-поведение.
