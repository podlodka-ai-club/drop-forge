## Context

DRO-40 (`Test1`) является минимальной Linear-задачей: есть ID, identifier и title, но нет описания и комментариев. Текущий оркестратор уже поддерживает передачу таких задач в proposal runner, поэтому это изменение рассматривается как OpenSpec smoke-test, а не как изменение runtime-поведения.

Основное ограничение: нельзя выводить новые продуктовые или технические требования из названия `Test1`. Proposal должен быть полезен для проверки процесса, но честно фиксировать, что исходная задача не содержит достаточного контекста для изменений в Go-коде.

## Goals / Non-Goals

**Goals:**

- Зафиксировать traceability к Linear task ID `e7119ecd-e524-4b32-ac12-b5dd4dc6db3c`, identifier `DRO-40` и title `Test1`.
- Создать проверяемую capability spec для smoke-проверки proposal generation.
- Подготовить tasks, которые можно выполнить без изменения runtime-кода и без добавления фиктивной функциональности.

**Non-Goals:**

- Не менять `internal/coreorch`, `internal/proposalrunner`, `internal/taskmanager`, CLI или конфигурацию.
- Не добавлять новые runtime-параметры в `.env.example`.
- Не обновлять `architecture.md`, потому что границы компонентов и orchestration flow остаются прежними.
- Не создавать требования к продуктовой функциональности, которых нет в Linear-задаче.

## Decisions

1. Создать новую capability `proposal-smoke-test`.

   Альтернатива: модифицировать `proposal-orchestration`. Она отклонена, потому что существующая спецификация уже требует валидный proposal input для задач без описания и комментариев, а DRO-40 не задает нового runtime-поведения.

2. Ограничить реализацию OpenSpec-артефактами.

   Альтернатива: добавить тест в Go-код для задачи без описания. Она отклонена для этой proposal-задачи, потому что соответствующие требования уже покрыты текущими specs; без нового behavior change такой тест был бы связан с существующей логикой, а не с DRO-40.

3. Считать отсутствие описания явным ограничением, а не пробелом, который нужно заполнить предположениями.

   Альтернатива: интерпретировать `Test1` как запрос на конкретную тестовую функциональность. Она отклонена, потому что это создало бы непроверяемые требования.

## Risks / Trade-offs

- Низкая прикладная ценность runtime-изменений -> mitigation: scope ограничен OpenSpec smoke-test и traceability.
- Возможное ожидание, что `Test1` должен был означать другую задачу -> mitigation: proposal явно фиксирует отсутствие исходных требований и оставляет дальнейшее поведение вне scope.
- Новый spec может выглядеть процессным, а не продуктовым -> mitigation: capability названа как smoke-test и не затрагивает архитектурные specs оркестратора.
