# AGENTS.md

## Назначение проекта

Этот репозиторий развивает оркестратор для кодинг-агентов.
Подход итеративный: начинаем с простого работающего решения, затем усложняем архитектуру только при реальной необходимости.

## Язык и стек

- Основной язык проекта: Go.
- Новые сервисы, утилиты и CLI по умолчанию писать на Go.
- Если нужен внешний инструмент, сначала обосновать, почему стандартной библиотеки Go недостаточно.

## Коммуникация

- Внутри проекта предпочтителен русский язык.
- Обсуждение задач, планы изменений и пояснения к реализации писать на русском.
- Спеки в `openspec` стараться писать на русском, если нет явной причины использовать английский.
- В коде допустим английский для идентификаторов, имен пакетов, API и технических терминов.

## Правила разработки

- Держать решения простыми и расширяемыми.
- Не добавлять абстракции заранее без подтвержденной необходимости.
- Перед изменением поведения сначала понять текущий поток данных и точки расширения.
- Следовать идиомам Go: маленькие пакеты, явные зависимости, простые интерфейсы.
- После изменений обязательно прогонять форматирование и тесты.

## Обязательные команды перед завершением задачи

Минимальный обязательный набор:

```bash
go fmt ./...
go test ./...
```

Если появятся дополнительные проверки, их нужно документировать и запускать вместе с тестами.

## Конфигурация

- Все runtime-параметры выносить в `.env`.
- В репозитории обязательно поддерживать актуальный `.env.example` без секретов.
- При добавлении новой переменной обновлять и `.env.example`, и код загрузки конфигурации.
- Нельзя хардкодить токены, ключи, URL окружений и прочие runtime-настройки.

Базовые ожидаемые файлы:

- `.env` — локальные значения для разработки.
- `.env.example` — шаблон переменных без секретов.

## Структура и рост проекта

- На раннем этапе предпочитать простую структуру каталогов.
- Выделять пакеты и модули только когда появляется явная ответственность или переиспользование.
- Для новых функций сначала собирать минимально рабочий сценарий, затем расширять наблюдаемость, отказоустойчивость и параллелизм.

## Работа со спеками

- Спеки хранить в `openspec`.
- Существенные изменения поведения сначала описывать в спеках и задачах.
- В спеках фиксировать цель, ограничения, сценарии и критерии приемки.

## Тестирование

- Любое изменение логики должно сопровождаться тестами или понятным объяснением, почему тест здесь пока неуместен.
- Предпочтение: table-driven tests для бизнес-логики и понятные интеграционные тесты для сценариев оркестрации.
- Если тесты не запускаются из-за инфраструктурного ограничения, это нужно явно указать в итоговом отчете.

## Практические договоренности

- Логику конфигурации держать централизованно.
- Ошибки оборачивать с контекстом.
- Публичное поведение CLI и API документировать по мере появления.
- Секреты, временные файлы и локальные артефакты не коммитить.
