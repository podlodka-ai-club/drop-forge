## Why

Сейчас оркестратор меняет статусы задач в Linear, но это действие остается внутренним побочным эффектом без расширяемой точки подписки. Для DRO-45 нужна архитектурная основа Event Driven событий, чтобы на смену статуса задачи можно было подписаться и отправлять уведомление в Telegram-чат.

## What Changes

- Добавить внутреннюю модель событий оркестратора и простой синхронный dispatcher, через который компоненты смогут публиковать доменные события.
- Публиковать событие смены статуса задачи после успешного `TaskManager.MoveTask`.
- Добавить Telegram-подписчик, который слушает события смены статуса задачи и отправляет сообщение в настроенный чат.
- Добавить runtime-конфигурацию Telegram и флаг включения уведомлений через `.env` / переменные окружения.
- Сохранить текущий orchestration flow: уведомления не должны менять порядок переходов статусов, запуск runner-ов или PR workflow.
- Не добавлять внешнюю очередь сообщений на этом этапе; расширение должно остаться локальным и минимальным.

## Capabilities

### New Capabilities
- `orchestration-events`: внутренние Event Driven события, dispatcher и contract подписчиков.
- `telegram-notifications`: отправка Telegram-сообщений по событиям смены статуса задачи.

### Modified Capabilities
- `linear-task-manager`: после успешной смены статуса задачи должен публиковаться event с идентичностью задачи и целевым статусом.

## Impact

- Код: `internal/taskmanager`, `internal/config`, `cmd/orchv3`, новый внутренний пакет для событий, новый внутренний пакет или адаптер Telegram-уведомлений.
- Конфигурация: новые переменные в `.env.example` без значений по умолчанию для включения Telegram, bot token и chat ID.
- Системы: Linear остается источником задач и местом смены статусов; Telegram Bot API используется только для доставки уведомлений.
- Тесты: unit-тесты dispatcher-а, публикации события из `TaskManager`, конфигурации Telegram и Telegram notifier-а через подменяемый HTTP client/server без реального Telegram API.
