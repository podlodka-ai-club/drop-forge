## Why

Текущий проект вырос из рабочего названия `orchv3`, которое теперь встречается в CLI, документации, конфигурации, именах временных директорий и пользовательских сообщениях. Перед дальнейшим развитием orchestration monitor нужно централизовать идентичность приложения и сделать управляемую миграцию конфигурации, чтобы новое название не ломало существующие локальные окружения.

## What Changes

- Ввести новое пользовательское имя продукта и заменить видимые упоминания `orchv3` в CLI, README, документации, логических service name'ах и временных workspace-prefix'ах.
- Предложить 3 веселых названия с ассоциацией "коты + подлодка":
  - `Catmarine`
  - `Purriscope`
  - `Meowrine`
- Рекомендованное название для реализации: `Catmarine`, потому что оно короткое, явно соединяет cat + submarine и хорошо читается как имя CLI/сервиса.
- Добавить централизованную модель application identity/config naming, чтобы дефолтное имя, service label и будущие пользовательские строки не расходились между пакетами.
- Выполнить миграцию переменных окружения на новый namespace и оставить старые ключи как deprecated aliases на переходный период.
- Обновить `.env.example`, README и релевантную документацию так, чтобы новые ключи были основными, а старые ключи были описаны как legacy aliases без секретов и значений по умолчанию.
- Добавить тесты конфигурационной миграции: новые ключи имеют приоритет, старые ключи продолжают работать, конфликтующие значения обрабатываются детерминированно.

## Capabilities

### New Capabilities
- `application-identity-config`: централизованное имя приложения, миграция runtime-конфигурации на новые env-ключи и совместимость со старым namespace.

### Modified Capabilities
- `project-readme`: README должен описывать новое имя, новую конфигурацию и правила миграции со старых env-ключей.

## Impact

- `go.mod`, `cmd/orchv3`, `internal/config`, `internal/proposalrunner`, `internal/applyrunner`, `internal/archiverunner`, `internal/steplog` и тесты, где имя приложения или env-key namespace влияет на публичное поведение.
- `.env.example`, README, `docs/*`, `architecture.md` при изменении пользовательского имени, CLI entrypoint или архитектурных границ конфигурации.
- Runtime configuration: новые env-ключи становятся предпочтительными; старые `PROPOSAL_*`/`APP_NAME=orchv3` сценарии должны продолжить работать до явного удаления в отдельной будущей change.
- Внешние зависимости не добавляются; миграция должна использовать существующую загрузку `.env` через `godotenv` и стандартную библиотеку Go.
