## Why

Текущее имя `orchv3` точно описывает техническую роль проекта, но выглядит как временное внутреннее название и слабо помогает запомнить продукт. Задача DRO-22 просит придумать более веселое название и логотип; комментарий про любовь к котикам задает понятное направление для дружелюбной, но не несерьезной айдентики.

## What Changes

- Ввести публичное бренд-имя проекта `Purrch` для README, документации и визуальной идентичности.
- Добавить логотип в cat-inspired стиле: кот-дирижер/оператор, который управляет coding-agent workflow.
- Обновить root README так, чтобы первое впечатление проекта начиналось с нового имени и логотипа, но сохраняло объяснение текущего CLI `orchv3`.
- Зафиксировать правила использования бренда: `Purrch` как человекочитаемое имя, `orchv3` как техническое имя CLI/модуля на этом этапе.
- Не переименовывать Go module, binary, env-переменные, package paths и публичные команды в рамках этой задачи.

## Capabilities

### New Capabilities

- `project-branding`: публичная бренд-идентичность проекта, включая выбранное название, логотип, расположение ассетов и правила использования имени в документации.

### Modified Capabilities

- `project-readme`: root README должен показывать новое бренд-имя и логотип как входную точку проекта, не скрывая текущий технический CLI `orchv3`.

## Impact

- Затронуты документация и статические ассеты: `README.md`, новая директория для бренд-ассетов, при необходимости краткий бренд-документ в `docs`.
- Runtime-поведение, Go API, CLI invocation, конфигурация и зависимости не меняются.
- `architecture.md` обновлять не требуется, потому что границы компонентов и orchestration flow не меняются.
