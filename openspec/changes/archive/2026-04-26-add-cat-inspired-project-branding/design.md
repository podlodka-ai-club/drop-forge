## Context

Проект сейчас представлен как `orchv3`: это точное, но техническое имя для Go CLI, который запускает proposal-runner workflow. Задача DRO-22 просит более веселое название и логотип, а комментарий про котиков дает направление для визуальной идеи.

Изменение не требует новой runtime-логики. Основная работа находится в документации и статических ассетах: выбрать имя, добавить логотип, обновить первое впечатление README и зафиксировать правила использования имени.

## Goals / Non-Goals

**Goals:**

- Ввести запоминаемое публичное имя проекта с cat-inspired характером.
- Добавить логотип, который можно использовать в README и документации без внешних сервисов.
- Сохранить техническую ясность: пользователь должен понимать, что текущая команда и Go module остаются `orchv3`.
- Свести изменение к документации и ассетам, чтобы не создавать миграцию для существующих скриптов.

**Non-Goals:**

- Переименование бинаря, Go module, package paths, env-переменных или GitHub repository.
- Добавление генераторов изображений, build-step для ассетов или новых зависимостей.
- Изменение orchestration flow, Linear-интеграции, `AgentExecutor`, `TaskManager` или конфигурации.
- Финальная бренд-система с большим набором вариантов, цветов и правил для внешнего маркетинга.

## Decisions

1. Публичное имя: `Purrch`.

   `Purrch` объединяет cat theme (`purr`) и сокращенный намек на orchestration (`orch`). Имя короткое, читается по-английски, не конфликтует с текущим техническим `orchv3` и подходит для README-заголовка.

   Рассмотренные альтернативы:

   - `CatOrch`: понятнее, но выглядит как прямое техническое имя без характера.
   - `Meowrch`: сильнее играет с котиками, но хуже читается и сложнее произносится.
   - `КотоОрк`: хорошо по-русски, но создает неоднозначность с `орк` и хуже подходит для CLI/README на смешанном русско-английском стеке.

2. Логотип: SVG-ассет в репозитории.

   Логотип должен быть векторным SVG, чтобы его можно было встроить в README без новых зависимостей, бинарных артефактов и build-step. Визуальный образ: кот-дирижер или кот-оператор, управляющий несколькими агентскими шагами, с аккуратной терминальной/оркестраторной отсылкой.

   Рассмотренные альтернативы:

   - PNG: проще для иллюстрации, но хуже масштабируется и создает бинарный артефакт.
   - Генерация изображения внешним сервисом: может дать более богатую картинку, но добавляет непрозрачный источник ассета и хуже воспроизводится.
   - Только текстовый логотип: быстрее, но не закрывает запрос про логотип.

3. Граница rename: бренд-имя меняется, техническое имя остается.

   README и документация должны показывать `Purrch` как название проекта, но явно объяснять, что текущий CLI запускается как `orchv3`. Это не ломает команды, env-переменные, package paths и существующие автоматизации.

   Рассмотренная альтернатива: сразу переименовать CLI и module. Это ближе к полному ребрендингу, но несет миграционный риск и выходит за рамки задачи про название и логотип.

4. Расположение артефактов: `docs/assets/logo.svg` и при необходимости `docs/branding.md`.

   `docs/assets/logo.svg` достаточно стабилен для ссылок из README. Короткий `docs/branding.md` нужен, если в реализации потребуется зафиксировать имя, смысл логотипа, ограничения и дальнейшие возможные rename-шаги отдельно от README.

## Risks / Trade-offs

- Риск: `Purrch` может быть воспринят как полное новое имя CLI. -> Mitigation: README должен рядом с названием явно указать текущую команду `orchv3`.
- Риск: SVG может выглядеть слишком декоративно для инфраструктурного проекта. -> Mitigation: визуальный стиль должен быть простым, техническим и читаемым в маленьком размере.
- Риск: частичный ребрендинг оставляет два имени. -> Mitigation: это осознанный этап; полный rename CLI/module можно сделать отдельным change после подтверждения бренда.
- Trade-off: выбран короткий SVG вместо детальной иллюстрации. Причина: он лучше подходит для репозитория, code review и долгосрочной поддержки без внешних инструментов.
