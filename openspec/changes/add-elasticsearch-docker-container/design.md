## Context

Сейчас репозиторий не содержит Docker-инфраструктуры для локального окружения и не описывает способ запуска Elasticsearch рядом с приложением. Из-за этого каждый разработчик вынужден поднимать Elastic вручную или работать без него, что делает локальный запуск непредсказуемым и усложняет дальнейшую интеграцию поиска или индексации.

Из proposal следует минимальная цель: добавить воспроизводимый dev-сценарий, в котором Elasticsearch стартует одной командой, публикует понятную точку подключения и не требует ручной системной установки. При этом проект сохраняет текущий прагматичный подход: минимум сущностей, минимум абстракций, только то, что нужно для локальной разработки.

## Goals / Non-Goals

**Goals:**

- Добавить единый Docker Compose сценарий для локального запуска Elasticsearch.
- Зафиксировать стабильные dev-настройки контейнера: single-node режим, опубликованный HTTP-порт, volume для данных и healthcheck.
- Подготовить приложение к подключению через централизованную env-конфигурацию, начиная с `ELASTICSEARCH_URL`.
- Документировать минимальный операционный поток: запуск, остановка, проверка доступности и очистка данных.

**Non-Goals:**

- Не добавлять кластер из нескольких Elasticsearch-нод.
- Не добавлять Kibana, Logstash, Beats и другие Elastic-компоненты.
- Не проектировать production deployment через Docker в рамках этого change.
- Не реализовывать бизнес-логику индексации или поисковые API, если их пока нет в приложении.

## Decisions

### Docker Compose как единственная точка локального запуска

Для первой версии достаточно одного compose-манифеста в корне репозитория. Он проще, чем набор shell-скриптов или отдельный orchestration layer, и уже отражает связь приложения с внешним локальным сервисом.

Альтернатива: документировать `docker run ...` без файла в репозитории. Это быстрее, но хуже поддерживается, сложнее расширяется и повышает риск расхождения параметров между разработчиками.

### Один контейнер Elasticsearch в dev single-node режиме

Локальный сценарий должен использовать официальный Elasticsearch image в single-node режиме. Это минимальная рабочая конфигурация для разработки и тестирования без кластерной координации.

Обязательные параметры контейнера:

- `discovery.type=single-node`;
- опубликованный HTTP-порт `9200`;
- именованный volume для сохранения данных между рестартами;
- healthcheck, который проверяет доступность HTTP endpoint контейнера.

Альтернатива: поднимать ephemeral-контейнер без volume. Это чуть проще, но создает лишние потери данных между перезапусками и усложняет отладку локальных сценариев.

### Локальную security-конфигурацию упростить

Для dev-окружения предлагается отключить обязательную аутентификацию и TLS в контейнере Elasticsearch, чтобы приложение могло подключаться по обычному `http://localhost:9200` без генерации сертификатов и учетных данных. Для локального сценария это оправдано, потому что цель change состоит в быстром предсказуемом старте среды, а не в production-hardening.

Альтернатива: оставлять security включенной по умолчанию, как в современных версиях Elasticsearch. Это ближе к production, но резко усложняет локальный bootstrap и увеличивает количество runtime-параметров, которые пока не нужны проекту.

### Конфигурацию подключения приложения централизовать через `.env`

Приложение должно читать endpoint Elasticsearch из env-конфигурации, начиная с `ELASTICSEARCH_URL`. Ключ необходимо добавить в `.env.example`, а загрузку и валидацию держать в `internal/config`, чтобы весь runtime-конфиг оставался централизованным.

Альтернатива: захардкодить `http://localhost:9200` в месте будущей интеграции. Это противоречит правилам репозитория и затруднит запуск в CI или при нестандартных портах.

### Документацию держать рядом с кодом проекта

Нужно добавить краткую инструкцию по локальному Elasticsearch либо в `README`, либо в существующий каталог `docs/`, в зависимости от принятого в репозитории формата. Документация должна покрывать:

- команду запуска;
- команду остановки;
- проверку health/status;
- очистку volume при необходимости полного сброса данных.

Альтернатива: оставить знания только в compose-файле. Это экономит время, но ухудшает discoverability и повышает цену поддержки.

## Risks / Trade-offs

- [Отключенная security в dev может отличаться от production] -> явно ограничить решение локальным окружением и не переносить эти настройки в production deployment.
- [Elasticsearch требует заметно больше памяти, чем другие локальные сервисы] -> зафиксировать умеренные JVM-лимиты для dev и задокументировать минимальные требования.
- [Версия image может повлиять на совместимость будущего клиента] -> использовать явно зафиксированный tag image, а не `latest`, и выбирать его при реализации вместе с кодом интеграции.
- [Сохранение volume может приводить к "грязным" данным между запусками] -> задокументировать команду полного сброса volume.

## Migration Plan

1. Добавить compose-манифест и env-переменные в репозиторий.
2. Обновить загрузку конфигурации приложения и `.env.example`.
3. Добавить документацию по запуску локального Elasticsearch.
4. Проверить сценарий "поднять контейнер -> дождаться healthcheck -> обратиться к endpoint из приложения".

Rollback: удалить compose-манифест, связанные env-переменные и документацию; код приложения должен продолжать работать без обязательного локального Elasticsearch.
