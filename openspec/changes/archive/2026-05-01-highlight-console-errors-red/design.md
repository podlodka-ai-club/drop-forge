## Context

CLI сейчас пишет события через `internal/steplog` в JSON Lines формате. `cmd/orchv3/buildLogger` направляет события в `stderr`, а при включенном Logstash дополнительно дублирует их в TCP sink через `io.MultiWriter`. Error-события отличаются только полем `type=error`, поэтому в интерактивной консоли они визуально не выделяются.

Ключевое ограничение: нельзя добавлять ANSI-последовательности в общий поток логов без разбора назначения writer-а, потому что это сломает контракт валидных JSON Lines для пайпов, тестов и внешних log sink-ов.

## Goals / Non-Goals

**Goals:**

- Подсвечивать красным целую строку консольного события с `type=error` при интерактивном запуске.
- Оставить `steplog.Logger` источником неизмененных JSON-событий.
- Гарантировать, что неинтерактивный вывод и Logstash/TCP sink получают исходные JSON Lines без ANSI-кодов.
- Покрыть поведение unit-тестами без зависимости от реального терминала.

**Non-Goals:**

- Не вводить новый формат логов или человеко-читаемый pretty-print вместо JSON.
- Не менять поля `time`, `service`, `module`, `type`, `message`.
- Не добавлять runtime-параметр в `.env`, если автоматической TTY-активации достаточно.
- Не раскрашивать info-события и не вводить цветовые темы.

## Decisions

1. Цвет применяется на уровне writer-а для интерактивной консоли, а не внутри `steplog.Logger`.

   `steplog.Logger` продолжает кодировать `Event` через `json.Encoder`. Новый helper получает уже готовые JSON Lines, буферизует их построчно, декодирует только минимально нужное поле `type`, и для `error` пишет строку как `ESC[31m` + line + `ESC[0m` + newline. Если строка не является JSON-событием, helper пишет ее без изменений.

   Альтернатива: добавить ANSI-коды прямо в `Logger.Errorf`. Она проще, но загрязняет все sink-и и ломает валидность JSON Lines.

2. Подсветка включается только для интерактивной TTY-копии stderr.

   В `cmd/orchv3/buildLogger` локальная консольная цель оборачивается цветным writer-ом только если writer является терминалом. При включенном Logstash итоговый writer остается `io.MultiWriter(coloredConsole, sink)`, поэтому logger отправляет один и тот же сырой JSON в обе цели, а ANSI появляются только при записи в консольную обертку.

   Альтернатива: включать цвет всегда. Это упростило бы UX в терминале, но испортило бы вывод при редиректе в файл, пайпах и тестах.

3. Проверка интерактивности должна быть тестируемой и без внешних зависимостей.

   Для production можно использовать стандартную библиотеку Go (`os.File.Stat` и `os.ModeCharDevice`). Для unit-тестов helper/решение получает явный флаг или небольшую функцию-предикат, чтобы проверить оба режима без настоящего TTY.

   Альтернатива: добавить `golang.org/x/term`. Для текущей задачи это лишняя зависимость, потому что достаточно простого определения character device.

## Risks / Trade-offs

- [Risk] ANSI-коды делают интерактивную stderr-копию невалидной JSON-строкой. → Mitigation: применять их только к TTY writer-у; все неинтерактивные writer-ы и внешние sink-и остаются raw JSON.
- [Risk] Цветная обертка может получить частичную строку. → Mitigation: буферизовать до newline и сбрасывать остаток при `Flush`/закрытии, если такой интерфейс будет нужен.
- [Risk] Терминал или пользовательское окружение может не поддерживать ANSI. → Mitigation: использовать базовый ANSI red/reset без тем и не добавлять обязательную конфигурацию; поведение не влияет на машинные потребители.
- [Risk] Парсинг каждой строки добавляет небольшую стоимость. → Mitigation: парсится только локальная интерактивная копия, а объем CLI-логов невелик по сравнению с выполняемыми внешними командами.
