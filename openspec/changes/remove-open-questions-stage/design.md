## Context

Сейчас proposal runner публикует комментарий в PR только если находит в markdown-артефактах секцию `Open Questions` или `Открытые вопросы`. Это связывает поведение runner с внутренним форматом skill `openspec-propose` и требует дополнительного обхода файлов после генерации proposal.

При этом сам `codex exec` уже умеет отдельно сохранять последнее сообщение агента через флаг `--output-last-message`. Для нового сценария это точнее отражает итог коммуникации Codex и убирает зависимость от структуры markdown-файлов.

## Goals / Non-Goals

**Goals:**
- Публиковать в PR последнее содержательное сообщение Codex после успешного создания pull request.
- Убрать зависимость runner от секций `Open Questions` внутри `openspec/changes/**/*.md`.
- Упростить локальный skill `openspec-propose`, чтобы он не требовал отдельный блок открытых вопросов в финальной сводке.
- Сохранить текущий поток логов proposal workflow и существующий happy path по созданию branch, commit, push и PR.

**Non-Goals:**
- Не менять формат самих OpenSpec-артефактов `proposal.md`, `design.md`, `tasks.md` и spec delta.
- Не вводить отдельный протокол обмена с Codex поверх JSONL-событий.
- Не менять механику создания PR body; меняется только источник дополнительного комментария.

## Decisions

### Захват последнего сообщения через `codex exec --output-last-message`

Runner будет передавать `codex exec` дополнительный путь к временному файлу, например внутри temp-dir рядом с clone workspace. После завершения команды runner читает этот файл, обрезает внешние пробелы и использует содержимое как тело PR comment.

Причины выбора:
- не нужно восстанавливать финальный ответ из JSONL-потока stdout;
- источник данных совпадает с тем, что сам Codex считает последним сообщением;
- сохраняется текущее потоковое логирование stdout/stderr без дополнительного парсинга.

Альтернатива: разбирать JSONL от `--json` и собирать последнее assistant-сообщение вручную. Это сложнее, сильнее зависит от схемы событий и увеличивает поверхность ошибок без реальной выгоды.

### Публиковать комментарий только при непустом последнем сообщении

Если файл с последним сообщением отсутствует, пустой или состоит только из whitespace, runner не создает пустой PR comment и всё равно возвращает URL pull request. Это повторяет текущую безопасную семантику "нет содержимого -> нет комментария", но уже без markdown-сканирования.

Альтернатива: считать отсутствие финального сообщения ошибкой workflow. Это слишком строго, потому что proposal и PR уже могут быть успешно созданы, а пустой комментарий не должен ломать основной результат.

### Упростить skill `openspec-propose` под новый contract

Локальный skill нужно обновить так, чтобы итоговый ответ больше не требовал отдельный блок `Open Questions`. Финальная сводка должна оставаться краткой: имя change, путь, созданные артефакты, готовность к `/opsx:apply`, принятые trade-offs при наличии.

Причина: runner теперь публикует весь последний ответ Codex целиком, поэтому отдельная стадия открытых вопросов больше не нужна и только провоцирует лишний шаблонный текст.

### Убрать markdown-сканирование из основного пути runner

Функции `CollectOpenQuestions`, `ExtractOpenQuestions` и сборка comment из списка вопросов больше не должны определять PR comment workflow. Их можно удалить вместе с тестами либо оставить только если они используются где-то еще, но текущий основной сценарий не должен зависеть от обхода `openspec/changes`.

Альтернатива: оставить старый поиск вопросов как fallback. Это смешивает два разных источника истины и делает поведение неочевидным: комментарий будет зависеть от того, что написал Codex, и одновременно от структуры markdown-файлов.

## Risks / Trade-offs

- [Финальное сообщение Codex может быть слишком длинным для комментария PR] -> публиковать текст как есть в первой версии, а ограничение длины и усечение добавить только при подтвержденной проблеме.
- [Codex может завершиться без last-message файла] -> трактовать это как отсутствие дополнительного комментария, не ломая успешное создание PR.
- [Изменение skill влияет на содержимое комментария] -> зафиксировать только минимальный формат финальной сводки и покрыть runner тестом на публикацию произвольного многострочного сообщения.
- [Удаление markdown-сканирования меняет тестовую поверхность] -> заменить существующие тесты open-questions сценариями для чтения last-message файла и пропуска пустого комментария.

## Migration Plan

1. Обновить skill `openspec-propose`, убрав требование отдельного блока `Open Questions` в финальной сводке.
2. Изменить `internal/proposalrunner` так, чтобы `codex exec` писал последнее сообщение в отдельный файл, а runner публиковал его как PR comment.
3. Удалить или вывести из основного пути код и тесты, завязанные на `CollectOpenQuestions`.
4. Обновить документацию proposal runner под новый workflow комментариев.
