## Context

Сейчас проект использует два разных подхода к логированию:

- `internal/steplog.Logger.Printf` пишет строки вида `[step] message` в переданный `io.Writer`.
- `cmd/orchv3/main.go` использует стандартный пакет `log` для fatal-ошибок и финального сообщения.

Из-за этого формат логов неоднородный, поле источника не является частью машинно-читаемого контракта, а тип события определяется только по тексту. Изменение затрагивает несколько модулей, но не требует внешних зависимостей: JSON-сериализация, время и запись в `io.Writer` покрываются стандартной библиотекой Go.

## Goals / Non-Goals

**Goals:**

- Ввести единый JSON Lines формат для логов приложения.
- Сделать обязательными поля `time`, `module`, `type`, `message`.
- Поддержать два типа событий: `info` и `error`.
- Сохранить простой вызов логгера из текущих workflow-шагов.
- Покрыть формат и ключевые сценарии unit-тестами.

**Non-Goals:**

- Не добавлять внешние logging-фреймворки.
- Не строить полноценную систему уровней, контекстов, trace/span id или sampling.
- Не менять runtime-конфигурацию и `.env.example`, так как новый формат не требует параметров окружения.
- Не менять бизнес-поведение proposal runner, кроме формата консольного вывода.

## Decisions

1. Использовать JSON Lines: одно событие лога равно одной JSON-строке с завершающим `\n`.

   Обоснование: такой формат легко читать построчно, он удобен для CI, shell pipeline и систем сбора логов. Альтернатива в виде единого JSON-массива не подходит для потокового вывода.

2. Зафиксировать минимальную схему события:

   ```json
   {"time":"2026-04-22T10:15:30.123456789Z","module":"proposalrunner","type":"info","message":"created temp dir /tmp/..."}
   ```

   Поле `time` пишется в UTC в формате RFC3339Nano. Поле `module` содержит стабильное имя модуля или внешнего шага (`cli`, `proposalrunner`, `temp`, `git`, `codex`, `github`). Поле `type` принимает только `info` или `error`. Поле `message` содержит человекочитаемый текст без структурных требований.

3. Развить `internal/steplog` в общий структурированный логгер.

   Текущий пакет уже централизует workflow-логи и принимает `io.Writer`, что удобно для тестов. Вместо добавления новой абстракции достаточно расширить его методами `Infof(module, format, args...)` и `Errorf(module, format, args...)`. Старый `Printf` можно удалить или оставить временным только как внутренний алиас на `Infof`, если это упростит миграцию в одном изменении.

4. CLI должен использовать тот же логгер вместо стандартного `log`.

   Fatal-сценарии должны сначала записывать JSON-событие с `type: "error"` и `module: "cli"`, затем завершаться с ненулевым кодом. Успешное сообщение с URL PR должно писаться как `type: "info"`.

5. Вывод внешних команд логируется как значение `message`.

   Для потокового stdout/stderr нужен небольшой writer-адаптер, который принимает байты от внешней команды, буферизует их до строки и пишет каждую строку как отдельное JSON-событие. Если Codex, git или gh возвращают многострочный вывод, каждая строка становится отдельным значением `message`. Для мест, где вывод нужен программе (`git status`, URL pull request), поток дополнительно пишется в `bytes.Buffer` через `io.MultiWriter`. Это сохраняет диагностическую информацию, не ломает JSON Lines контракт и не требует менять интерфейс `commandrunner.Runner`.

## Risks / Trade-offs

- Риск: существующие тесты или пользователи ожидают старый текстовый формат `[step] message` -> Митигация: обновить тесты и явно зафиксировать изменение в spec как изменение публичного консольного формата.
- Риск: слишком свободное поле `module` может привести к разным названиям одного источника -> Митигация: определить стабильный минимальный набор модулей в тестах для текущих точек логирования.
- Риск: многострочные сообщения станут менее удобны для ручного чтения -> Митигация: JSON Lines важнее для машинной обработки; текст остается доступным после JSON-декодирования.
- Риск: ошибки записи в `io.Writer` сейчас могут игнорироваться -> Митигация: для текущего CLI достаточно best-effort логирования, но реализация должна быть простой для тестирования через writer, который возвращает ошибку, если потребуется расширение.

## Migration Plan

1. Обновить `internal/steplog` для записи JSON-событий.
2. Перевести `internal/proposalrunner` с `Printf` на `Infof`/`Errorf` или совместимый структурированный метод.
3. Перевести `cmd/orchv3` со стандартного `log` на `steplog`.
4. Обновить и добавить тесты на JSON-формат, обязательные поля и типы событий.
5. Запустить `go fmt ./...` и `go test ./...`.
