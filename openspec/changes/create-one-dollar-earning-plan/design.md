## Context

Текущий `orchv3` уже умеет брать Linear-задачи, создавать OpenSpec proposal PR и переводить задачу на review. DRO-29 не требует изменения этого workflow: формулировка просит план, а не новую функциональность CLI. Поэтому реализация должна быть минимальной: один документационный артефакт, который можно проверить ревьюом и затем использовать вручную.

План должен опираться на фактический контекст проекта: это инструмент для proposal-stage orchestration кодинг-агентов, а не готовый коммерческий SaaS. Значит, первый `$1` реалистичнее получить через ручное service offering или paid validation вокруг текущей экспертизы, чем через автоматизированный платежный поток внутри приложения.

## Goals / Non-Goals

**Goals:**

- Создать понятный markdown-план заработка первого `$1`.
- Сфокусировать план на минимальном ручном действии, которое можно выполнить без разработки платежной инфраструктуры.
- Зафиксировать проверяемые критерии успеха: кто платит, за что платит, как подтверждается оплата или обязательство оплатить.
- Оставить документ пригодным для последующих итераций, если появится реальная обратная связь.

**Non-Goals:**

- Не добавлять платежные интеграции, лендинги, CRM или внешние зависимости.
- Не менять orchestration flow, Linear-интеграцию, CLI-команды или конфигурацию.
- Не обещать гарантированный доход и не оформлять финансовую консультацию.
- Не строить полноценную go-to-market стратегию для продукта.

## Decisions

- Хранить план в `docs/one-dollar-earning-plan.md`.
  - Альтернатива: добавить раздел в `README.md`.
  - Причина: README уже описывает проект и запуск CLI; коммерческий план является отдельным рабочим артефактом и не должен размывать входную документацию.

- Делать план ручным и проверяемым, а не автоматизировать монетизацию в коде.
  - Альтернатива: добавить CLI-команду или интеграцию с платежной системой.
  - Причина: задача просит план, а первый `$1` можно валидировать быстрее через ручной outreach, invoice/link на оплату или письменное paid commitment.

- Связать план с текущей ценностью `orchv3`: ускорение подготовки OpenSpec proposal по Linear-задачам.
  - Альтернатива: описать универсальные способы заработать `$1`.
  - Причина: артефакт должен быть полезен именно этому репозиторию и текущему направлению проекта.

## Risks / Trade-offs

- Слишком общий план может не привести к оплате -> добавить конкретное предложение, целевой профиль покупателя, outreach-скрипт и критерий done.
- Слишком продуктовый план потребует лишней разработки -> ограничить scope ручной продажей и delivery без новых систем.
- Непроверяемые метрики осложнят ревью -> считать успехом полученный `$1`, оплаченный инвойс/платежную ссылку или явное письменное согласие оплатить конкретный deliverable.
- Документ может устареть после изменения позиционирования проекта -> держать план в `docs/` как версионируемый markdown и обновлять отдельными changes.
