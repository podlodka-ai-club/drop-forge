## Context

Проект уже документирует подтвержденный runtime-сценарий через корневой `README.md` и `docs/proposal-runner.md`, но эти материалы ориентированы в первую очередь на запуск CLI и внешние prerequisites. Для нового участника все еще нет отдельного документа, который в одном месте объясняет:

- зачем существует `orchv3`;
- какой пользовательский сценарий реализован сейчас;
- как проходит основной поток от входной задачи до PR URL;
- где в кодовой базе лежат ключевые части логики;
- какие ограничения у текущей реализации важно понимать заранее.

Изменение относится только к документации и навигации по репозиторию. Источниками правды остаются `cmd/orchv3/main.go`, `internal/config/config.go`, `internal/proposalrunner`, `.env.example`, `README.md` и `docs/proposal-runner.md`.

## Goals / Non-Goals

**Goals:**

- Добавить отдельный обзорный документ проекта как первую точку входа для чтения.
- Синтезировать в этом документе текущее назначение проекта, основной поток работы и ключевые пакеты.
- Явно описать текущие границы продукта: что уже поддерживается и что пока не входит в scope.
- Обеспечить discoverability обзорного документа из корневого `README.md`.

**Non-Goals:**

- Не менять runtime-поведение `orchv3`, конфигурацию или внешний workflow.
- Не дублировать целиком `docs/proposal-runner.md`; подробные операционные детали остаются в нем.
- Не документировать гипотетические будущие режимы работы, которых нет в коде.
- Не вводить новую сложную структуру документации вне одного дополнительного обзорного документа и ссылки на него из README.

## Decisions

### 1. Вынести обзор проекта в отдельный документ `docs/project-overview.md`

Отдельный документ лучше подходит для ответа на вопрос "что это за проект" без перегрузки корневого `README.md`. README остается коротким входом в репозиторий и местом для команд запуска, а обзорный документ берет на себя объяснение контекста и структуры.

Альтернатива: расширить `README.md` до полного архитектурного описания. Это сделает корневой документ слишком длинным и усложнит поддержку, потому что README уже несет роль quick start.

### 2. Строить обзор вокруг текущего end-to-end потока, а не вокруг всех файлов подряд

Документ должен описывать проект через реальный пользовательский сценарий:

1. CLI читает описание задачи из аргументов или `stdin`.
2. Загружается конфигурация из environment и `.env`.
3. `proposalrunner` создает временный клон и запускает `codex exec`.
4. После изменений выполняются `git` и `gh` шаги, а пользователю возвращается PR URL.

Такой порядок помогает понять систему быстрее, чем длинный листинг пакетов без связи с потоком данных.

Альтернатива: дать только перечень каталогов. Это полезно для навигации, но не объясняет поведение системы.

### 3. Обзорный документ должен явно фиксировать ограничения текущего scope

В документ стоит включить отдельный блок с ограничениями текущей реализации: проект пока представляет собой Go CLI для proposal-runner workflow, не предоставляет HTTP API, не описывает deployment-сценарии и зависит от внешних `git`/`codex`/`gh`.

Альтернатива: ограничиться позитивным описанием возможностей. Это повышает риск неверных ожиданий у нового читателя.

### 4. README обязан ссылаться на обзорный документ как на основной материал для знакомства

Без явной ссылки новый документ легко останется "мертвым" артефактом в `docs/`. Поэтому корневой README должен направлять читателя к нему при первом знакомстве с репозиторием.

Альтернатива: полагаться только на раздел `docs/` и ручной поиск. Это делает навигацию хуже и не решает исходную проблему.

## Risks / Trade-offs

- [Обзорный документ начнет расходиться с кодом] -> опираться только на текущие исходники и существующую документацию как источники правды, не описывать неподтвержденные сценарии.
- [Появится дублирование с README и `docs/proposal-runner.md`] -> оставить в обзоре высокий уровень, а за операционными деталями отсылать в существующие документы.
- [Документ получится слишком абстрактным] -> включить конкретный end-to-end поток, список ключевых пакетов и явные ограничения.
- [README станет избыточным] -> добавить только ссылку и краткое позиционирование, не переносить туда весь обзор.

## Migration Plan

1. Создать `docs/project-overview.md` с обзором назначения проекта, потока выполнения, ключевых пакетов, конфигурации и ограничений.
2. Обновить `README.md`, чтобы он ссылался на новый обзорный документ как на основной entrypoint для знакомства.
3. Проверить, что описание не противоречит `cmd/orchv3/main.go`, `internal/config/config.go`, `internal/proposalrunner`, `.env.example` и `docs/proposal-runner.md`.
4. Прогнать стандартные проверки репозитория, даже несмотря на документирующий характер изменения.

Rollback тривиален: удалить `docs/project-overview.md` и откатить ссылку из `README.md`.
