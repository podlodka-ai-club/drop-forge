## Context

DRO-39 просит провести ресерч конкурентных проектов и подсмотреть идеи у них. Источник проектов задан как GitHub-страница организации `podlodka-ai-club`; на момент подготовки proposal страница показывает 15 публичных репозиториев, но implementation должен перечитать актуальный список, потому что состав репозиториев может измениться.

В текущем проекте уже есть архитектурная рамка вокруг `CoreOrch`, `TaskManager`, `AgentExecutor`, `GitManager` и `Logger`. Исследование должно помочь выбрать будущие улучшения для этих границ, но не должно смешивать discovery с немедленным изменением runtime-поведения.

## Goals / Non-Goals

**Goals:**

- Собрать воспроизводимый markdown-отчет с источниками, датой исследования и актуальным списком изученных репозиториев.
- Для каждого релевантного проекта зафиксировать назначение, стек, структуру, ключевые workflow, интеграции и идеи, применимые к оркестратору.
- Сравнить проекты по единым критериям, чтобы выводы были проверяемыми, а не набором субъективных заметок.
- Подготовить shortlist идей с оценкой пользы, сложности, рисков и рекомендуемого порядка дальнейших OpenSpec changes.

**Non-Goals:**

- Не реализовывать найденные идеи в рамках этого change.
- Не добавлять новые runtime-зависимости, конфигурацию или orchestration stage.
- Не обновлять `architecture.md`, если итогом остается только исследовательский документ.
- Не копировать код конкурентов; исследование фиксирует идеи, паттерны и продуктовые решения.

## Decisions

1. Исследование оформляется как документация, а не как кодовая фича.

   Альтернатива: сразу добавить отдельный research-stage в оркестратор. Это преждевременно: текущая задача просит провести ресерч, а не автоматизировать повторяемый процесс. Документ даст входные данные для следующих решений без расширения runtime.

2. Список репозиториев фиксируется в отчете вместе с датой и источником.

   Альтернатива: зашить текущие 15 репозиториев в OpenSpec. Это сделало бы proposal устаревающим; GitHub-страница уже является live-источником, поэтому implementation должен перечитать ее или GitHub API на момент выполнения.

3. Анализ идет по единой матрице критериев.

   Альтернатива: свободные заметки по каждому проекту. Свободный формат быстрее, но хуже сравнивается и усложняет выбор идей для backlog. Матрица удерживает исследование в полезном для принятия решений виде.

4. Идеи для внедрения отделяются от фактов исследования.

   Альтернатива: смешать наблюдения и рекомендации в одном списке. Разделение снижает риск принять непроверенную догадку за подтвержденную возможность и упрощает последующую декомпозицию в OpenSpec changes.

## Risks / Trade-offs

- Неполные README или закрытая логика проектов -> смотреть не только README, но и структуру каталогов, issues, PR, конфиги CI, Docker/compose-файлы и тесты.
- Список репозиториев изменится между proposal и implementation -> в отчете указывать дату исследования, источник и фактически изученный список.
- Исследование уйдет в слишком широкий обзор -> ограничить критерии темами, релевантными оркестратору: task flow, agent execution, git/PR workflow, observability, configuration, UX/CLI и тестирование.
- Рекомендации окажутся слишком крупными для immediate implementation -> оформлять их как shortlist с оценкой сложности и предлагать отдельные OpenSpec changes.
