## Context

В репозитории уже есть несколько источников пользовательской и инженерной документации: корневой `README.md`, документы в `docs/`, `architecture.md`, `.env.example` и актуальные OpenSpec specs. Задача DRO-33 сформулирована как проверка актуальности документации, поэтому основная ценность изменения — не новый runtime-код, а воспроизводимый аудит документации относительно текущего состояния Go-кода и конфигурации.

Текущий код уже реализует proposal orchestration pass, Linear-facing `TaskManager`, proposal runner через `AgentExecutor`, структурное логирование и ELK demo. Документация должна описывать именно эти возможности, не обещая неподдержанные режимы вроде постоянного scheduler/polling loop или автоматического retry/locking.

## Goals / Non-Goals

**Goals:**

- Сверить публичные документы с текущими CLI-режимами, runtime-конфигурацией, архитектурным маппингом и OpenSpec requirements.
- Исправить найденные расхождения в документации минимальными изменениями.
- Зафиксировать критерии, по которым будущие изменения смогут проверять актуальность документации.
- Сохранить `.env.example` как список ключей без секретов и без значений по умолчанию.

**Non-Goals:**

- Не добавлять новый CLI-режим, scheduler, retry/locking или другую runtime-функциональность.
- Не переписывать документацию в новый формат и не менять структуру каталогов без выявленной необходимости.
- Не менять код только ради соответствия документации, если корректнее обновить устаревший текст.
- Не документировать внутренние детали, которые не являются публичным контрактом или устойчивой архитектурной границей.

## Decisions

1. Проверять документацию через явный чеклист источников правды.

   Источники правды для аудита: `cmd/orchv3`, `internal/config`, `internal/coreorch`, `internal/proposalrunner`, `internal/taskmanager`, `internal/steplog`, `deploy/`, `.env.example` и актуальные `openspec/specs`. Альтернатива — ручное чтение только `README.md`, но это не покрывает расхождения в специализированных документах.

2. Считать результатом задачи исправленную документацию или явно зафиксированное отсутствие расхождений.

   Если документ уже актуален, его не нужно менять ради churn. Альтернатива — вносить косметические обновления в каждый документ, но это увеличивает шум и не улучшает проверяемость.

3. Не обновлять `architecture.md`, если аудит не выявит изменения границ ответственности.

   Правила проекта требуют менять `architecture.md` только при нетривиальных архитектурных изменениях. В рамках этой задачи ожидается проверка и возможная коррекция текста, а не перенос ответственности между компонентами.

4. Не вводить автоматический documentation linter в этой итерации.

   Текущая задача просит проверить актуальность, а не построить инструмент. Автоматизация может появиться позже, когда повторяющиеся расхождения станут понятны.

## Risks / Trade-offs

- Неполная формулировка Linear-задачи может оставить спорным объем аудита → scope ограничен существующими публичными документами и runtime-контрактами, видимыми в коде.
- Проверка остается ручной и зависит от внимательности исполнителя → spec задает обязательные сценарии сверки, а tasks фиксируют конкретный чеклист.
- Документация может снова устареть после будущих code changes → OpenSpec capability делает актуальность документации явным критерием для последующих изменений.
- Возможны расхождения между OpenSpec specs и фактической реализацией → при аудите приоритет имеет текущий код для пользовательских инструкций, а найденные spec-расхождения фиксируются как документационные правки или отдельные follow-up задачи.
