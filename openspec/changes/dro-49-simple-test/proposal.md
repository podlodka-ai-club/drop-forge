## Why

Linear-задача DRO-49 создана как минимальная проверка контура proposal orchestration: у нее есть только идентификатор и заголовок `Просто тест`, без описания и комментариев. Такой низкоконтекстный вход должен обрабатываться предсказуемо, чтобы оркестратор можно было проверять smoke-сценарием без ручного добавления фиктивного описания.

## What Changes

- Уточнить поведение proposal orchestration для задач с минимальным контекстом из Linear.
- Требовать, чтобы proposal runner получал валидный вход и мог создать OpenSpec proposal даже при отсутствии описания и комментариев.
- Зафиксировать, что сгенерированный proposal должен явно показывать source Linear task, принятые допущения и отсутствие открытого контекста, вместо того чтобы выдумывать полноценную продуктовую фичу.
- Добавить тестовый сценарий для задачи вида DRO-49: идентификатор и заголовок есть, описание и комментарии отсутствуют.

## Capabilities

### New Capabilities

- Нет.

### Modified Capabilities

- `proposal-orchestration`: уточняется обработка низкоконтекстных Linear-задач как smoke/validation proposal без отказа исполнения.

## Impact

- Затрагивает контракт proposal orchestration и тесты вокруг подготовки `ProposalInput`.
- Не требует новых зависимостей, переменных окружения или изменения публичного CLI.
- Реализация должна оставаться локальной для маршрута proposal orchestration и его fake-зависимостей в тестах.
