## Context

Кодовая база уже содержит несколько stage-specific runner'ов с близкой структурой: `proposalrunner`, `applyrunner`, `archiverunner` и частично `reviewrunner`. Часть повторов является полезной симметрией стадий: разные prompt'ы, commit messages, state transitions и result contracts остаются явными. Другая часть выглядит механической: одинаковые `AgentExecutionInput`, `runLoggedCommand`, `codexArgs`, writer helpers и похожие тестовые fake/builders.

Задача DRO-58 сформулирована как проверка дублей и предложение исправлений, поэтому первый результат должен быть проверяемым инженерным отчетом, а не крупной абстракцией. Исправлять в рамках этого изменения стоит только те дубли, где общий код уже имеет естественную ответственность и не скрывает различия стадий.

## Goals / Non-Goals

**Goals:**

- Найти и описать значимые дубли в production Go-коде и тестах.
- Разделить повторы на механические, stage-specific, тестовые и намеренно оставленные.
- Предложить для каждого значимого кластера конкретное действие: исправить сейчас, отложить до следующей боли или оставить как явную симметрию.
- Выполнить низкорисковые исправления, если они уменьшают механическое копирование без изменения публичного поведения.
- Проверить изменения через `go fmt ./...` и `go test ./...`.

**Non-Goals:**

- Не добавлять обязательный внешний duplicate detector в build/test pipeline.
- Не менять Linear workflow, git workflow, OpenSpec workflow, CLI contract или runtime-конфигурацию.
- Не объединять все runner'ы в универсальный stage framework только ради уменьшения количества строк.
- Не переписывать существующие тесты масштабно, если дубли в них помогают локально читать сценарии.

## Decisions

### D1. Аудит как отдельный документ

Создать `docs/duplicate-code-audit.md` с таблицей кластеров дублей. Для каждого кластера фиксировать файлы, тип дубля, риск, рекомендуемое действие и статус решения.

Альтернатива: сразу править найденные повторы без отдельного отчета. Отброшено, потому что задача просит именно проверить код и предложить исправления; без отчета будет сложно понять, какие дубли осознанно оставлены.

### D2. Исправлять только механические дубли с очевидной общей ответственностью

Кандидаты первого уровня:

- одинаковый `runLoggedCommand` в runner-пакетах можно вынести в `internal/commandrunner`, так как пакет уже знает о `steplog` и внешних командах;
- общие helpers для writer fallback или Codex argv можно извлекать только если место назначения не создает циклических зависимостей и не размывает stage-specific prompt logic;
- тестовые builders/fakes извлекать только внутри пакета или в уже существующую тестовую область, если это не ухудшает читаемость сценариев.

Альтернатива: ввести новый общий пакет `internal/stagerunner` для всех runner lifecycle. Отложено: apply/archive/proposal похожи, но имеют разные contracts, PR behavior и final-message semantics; такая абстракция сейчас может спрятать важные различия.

### D3. Stage symmetry не считать ошибкой автоматически

Похожие `ApplyInput` и `ArchiveInput`, validation flows, commit message builders и prompt builders считаются кандидатами на документированное отложенное исправление, если их объединение требует generic stage model или приводит к менее явным ошибкам. Аудит должен явно отмечать, почему такой дубль оставлен или почему его стоит исправить.

Альтернатива: удалять любой повтор по совпадению строк. Отброшено: в оркестраторе повтор иногда полезен как локальный contract каждой стадии.

### D4. Внешние инструменты не становятся частью проекта

Можно использовать разовые локальные команды (`rg`, `go test`, `go list`, простые shell-сравнения) для аудита. Если будет предложен внешний detector, он должен быть описан в отчете как необязательный вариант с обоснованием, но не добавлен в `go.mod`, CI или обязательные команды без отдельного решения.

Альтернатива: добавить специализированный duplicate detector в зависимости. Отложено до момента, когда ручной аудит перестанет масштабироваться.

## Risks / Trade-offs

- [Переизвлечение скроет различия стадий] -> Исправлять только механические дубли и оставлять stage contracts явными.
- [Отчет устареет после следующих изменений] -> Хранить в отчете дату, scope и статус каждого решения; обновлять его при повторном аудите.
- [Тестовые дубли будут выглядеть как лишний шум] -> Оценивать тестовые повторы отдельно: исправлять только builders/fakes, которые реально мешают поддержке.
- [Разовый аудит не гарантирует постоянную защиту от дублей] -> Зафиксировать это как принятый trade-off; автоматизацию добавлять отдельным изменением при повторяющейся проблеме.

## Migration Plan

1. Провести ручной аудит production и test-кода с фокусом на runner-пакеты, `coreorch`, `reviewrunner` и общие helper'ы.
2. Создать `docs/duplicate-code-audit.md` с классификацией и рекомендациями.
3. Реализовать только подтвержденные низкорисковые refactorings из отчета.
4. Обновить или добавить unit tests для затронутых helper'ов и runner'ов.
5. Обновить `architecture.md` только при появлении нового общего пакета или изменении границ ответственности.
6. Запустить `go fmt ./...` и `go test ./...`.

Rollback: revert refactoring commits; отчет можно оставить как диагностический артефакт, если он не содержит неверных утверждений.

## Open Questions

  - Recommended option: не добавлять в этом изменении. Сначала проверить, есть ли повторяющаяся боль после ручного отчета и точечных исправлений.
