## Context

Текущий `coreorch.Orchestrator.RunProposalsOnce` загружает управляемые Linear-задачи одним списком, маршрутизирует их по состояниям proposal/apply/archive и выполняет каждую подходящую задачу прямо в цикле. Это сохраняет простой порядок мутаций, но один долгий runner блокирует все следующие задачи из того же pass.

Executor'ы proposal, apply и archive уже изолируют тяжелую работу в отдельных временных checkout'ах и тестируются через интерфейсы. Это позволяет распараллелить запуск на уровне orchestration stage без изменения контрактов runner'ов.

## Goals / Non-Goals

**Goals:**

- Запускать независимые proposal/apply/archive task runs в отдельных горутинах.
- Ограничить параллелизм дефолтным лимитом `2`, чтобы выполнить требование "одновременно делать две задачи".
- Сохранить текущие переходы Linear state внутри каждого task run.
- Не останавливать уже запущенные task runs из-за ошибки другой задачи.
- Вернуть из pass агрегированную ошибку, если одна или несколько задач завершились неуспешно.
- Сохранить тестируемость через fake `TaskManager` и fake runner'ы.

**Non-Goals:**

- Не менять внутреннюю реализацию proposal/apply/archive runner'ов.
- Не добавлять распределенную очередь, внешний worker pool или персистентный scheduler.
- Не вводить межпроцессную блокировку от второго запущенного экземпляра оркестратора.
- Не менять публичные Linear states и не добавлять новые workflow-состояния.

## Decisions

1. Добавить лимит параллелизма в `coreorch.Config`.

   `MaxConcurrentTasks` должен управлять числом одновременно выполняемых task runs. Если значение не задано или равно `0`, orchestration runtime использует дефолт `2`. Отрицательное значение считается ошибкой валидации. Альтернатива с хардкодом `2` проще, но быстро создаст новый кодовый change при необходимости временно снизить или поднять лимит.

2. Реализовать worker pool на стандартных горутинах и каналах.

   `RunProposalsOnce` после загрузки задач формирует список `taskRun` для состояний ready-to-propose, ready-to-code и ready-to-archive, логирует skipped задачи, затем запускает не больше `MaxConcurrentTasks` workers. Каждый worker вызывает существующий `process*Task` для конкретного типа. Альтернатива с отдельной горутиной на каждую задачу и semaphore тоже допустима, но worker pool проще тестировать по максимальному числу активных runner'ов.

3. Сохранять порядок внутри одной задачи, но не глобальный порядок списка.

   Для каждой задачи остаются прежние шаги: build input, move to in-progress, runner, финальный transition/PR attachment. Между разными задачами порядок мутаций больше не гарантируется. Это отражает цель конкурентности и убирает искусственное ожидание долгой операции.

4. Не отменять siblings при ошибке task run.

   Ошибка одной задачи записывается в общий результат, но уже начатые и еще ожидающие слот задачи продолжают выполняться в рамках pass. Это лучше подходит для Linear workflow: независимая задача не должна простаивать из-за сбоя другой. Альтернатива с cancel-on-first-error экономит ресурсы, но хуже соответствует требованию выполнять две задачи одновременно.

5. Агрегировать ошибки стандартными средствами Go.

   После завершения workers pass возвращает `errors.Join(...)` с контекстными ошибками из `process*Task`. Это не требует новой зависимости и сохраняет возможность проверять конкретные причины через `errors.Is/As` там, где это понадобится.

6. Конфигурировать лимит через `.env` только на runtime-уровне.

   Добавить `ORCH_MAX_CONCURRENT_TASKS` в config loading и `.env.example` без значения по умолчанию. Кодовый дефолт остается `2`. Это соответствует существующему подходу к runtime-параметрам и не требует отдельного пакета конфигурации.

## Risks / Trade-offs

- Одновременные вызовы `TaskManager.MoveTask` и `TaskManager.AddPR` могут раскрыть потоконебезопасность fake или реального клиента -> использовать потокобезопасные тестовые fake'и и проверить, что production client не хранит изменяемое состояние без защиты.
- Логи разных задач будут перемешиваться -> включать task ID, identifier, route и событие task run start/done/error в каждую важную запись.
- Порядок мутаций между задачами станет недетерминированным -> обновить тесты так, чтобы они проверяли per-task порядок и лимит concurrency, а не общий порядок всего списка.
- `errors.Join` меняет текст итоговой ошибки при нескольких отказах -> тестировать наличие контекста каждой отказавшей задачи, а не полное строковое совпадение.
