## Context

Проект находится на раннем этапе и пока запускает CLI-приложение на Go с централизованной загрузкой конфигурации из env-переменных. Логи сейчас пишутся стандартным пакетом `log` в stdout, а локальной инфраструктуры для хранения и поиска логов нет. Изменение должно добавить минимальный Elasticsearch для разработки, не превращая это в полноценный observability stack.

В репозитории пока нет Docker Compose конфигурации, поэтому новый контейнер Elastic станет первым описанным локальным инфраструктурным сервисом. Все runtime-параметры должны быть отражены в `.env.example` без значений по умолчанию, а логика чтения переменных должна оставаться в `internal/config`.

## Goals / Non-Goals

**Goals:**

- Добавить локальный Docker Compose сервис Elasticsearch для хранения логов.
- Сделать запуск воспроизводимым через стандартный `docker compose up`.
- Добавить healthcheck, чтобы разработчик видел готовность Elastic.
- Хранить данные Elastic в named volume, чтобы перезапуск контейнера не удалял логи.
- Добавить в конфигурацию приложения параметры подключения к Elastic: URL и имя индекса логов.
- Покрыть загрузку новых параметров тестами конфигурации.
- Кратко задокументировать запуск и проверку доступности.

**Non-Goals:**

- Не реализовывать отправку логов приложения в Elasticsearch.
- Не добавлять Kibana, Logstash, Beats, OpenTelemetry Collector или отдельный ingestion pipeline.
- Не проектировать production-ready кластер Elasticsearch.
- Не вводить аутентификацию Elastic для локального dev-контейнера.
- Не добавлять внешние Go-зависимости, пока приложение только хранит параметры подключения.

## Decisions

1. Использовать `docker-compose.yml` в корне репозитория.

   Это самый простой способ описать локальный инфраструктурный сервис рядом с приложением. Альтернативой был бы отдельный shell-скрипт с `docker run`, но он хуже масштабируется при добавлении следующих сервисов и сложнее документирует volume, ports и healthcheck.

2. Поднять один сервис `elasticsearch` в режиме `single-node`.

   Для разработки и хранения логов одного узла достаточно. Альтернатива в виде multi-node кластера добавила бы настройки discovery, ресурсы и операционную сложность без текущей пользы.

3. Отключить security features в локальном контейнере.

   Локальный контейнер нужен для быстрого dev-сценария и не должен требовать ручной настройки паролей или сертификатов. Это должно быть явно ограничено документацией как local-only режим. Альтернатива с включенной безопасностью лучше ближе к production, но сейчас усложнит старт и потребует секретов, которые нельзя хранить в `.env.example`.

4. Использовать named volume для данных Elastic.

   Логи должны переживать перезапуск контейнера. Bind mount в локальную папку сделал бы права файловой системы и `.gitignore` более хрупкими, поэтому named volume проще и надежнее для первого шага.

5. Экспортировать порт `9200` на host и добавить healthcheck через HTTP endpoint Elastic.

   Это дает простой способ проверки: `curl http://localhost:9200/_cluster/health`. Альтернатива без публикации порта подходит только для контейнерной сети, но текущий CLI запускается напрямую на host.

6. Добавить параметры `ELASTICSEARCH_URL` и `ELASTICSEARCH_LOG_INDEX` в конфигурацию Go.

   URL нужен приложению для будущей отправки логов и health checks, имя индекса фиксирует ожидаемое место хранения логов. Переменные читаются централизованно в `internal/config`, а `.env.example` содержит только ключи без значений, как требует проект.

## Risks / Trade-offs

- Локальный Elastic может потребовать больше памяти, чем текущее приложение → в Compose нужно задать умеренные JVM options и задокументировать, что сервис предназначен для разработки.
- Отключенная security конфигурация небезопасна вне локальной машины → документация должна явно помечать Compose как local-only и не использовать такие настройки для production.
- Публикация порта `9200` может конфликтовать с уже запущенным Elastic → оставить порт параметризуемым только если это потребуется позже; на первом шаге документировать стандартный порт.
- Добавление только контейнера без ingestion не даст логам автоматически попадать в Elastic → specs и задачи должны явно ограничить область изменения, чтобы не создавать ложного ожидания.
