## Context

`orchv3` сейчас работает как proposal-runner CLI: получает текст задачи, при непустом вводе запускает внешний workflow через `git`, `codex` и `gh`, а при пустом вводе только пишет стартовый лог. Такой контракт удобен для изменений в целевом репозитории, но не покрывает информационные запросы о текущем проекте, хотя в репозитории уже есть локальные источники правды для ответа: `README.md`, `docs/proposal-runner.md`, `.env.example`, а также код CLI и config-пакета.

Новый режим должен решить две задачи одновременно:

- ответить на запрос вида `Расскажи мне о проекте` без сетевых зависимостей и без создания PR;
- не сломать существующий сценарий proposal-runner для входов, которые действительно описывают изменение.

## Goals / Non-Goals

**Goals:**

- Добавить детерминированный обзорный режим для явных информационных запросов о проекте.
- Выполнять обзорный режим полностью локально, без временного clone workspace и без внешних CLI.
- Возвращать краткий, полезный и проверяемый обзор проекта по текущему состоянию репозитория.
- Сохранить существующее proposal-поведение для задач на изменение.
- Сделать поведение покрываемым unit-тестами без сети и без реальных внешних процессов.

**Non-Goals:**

- Не добавлять общий NL-классификатор произвольных пользовательских намерений.
- Не делегировать обзор проекта в Codex или другой внешний агент.
- Не превращать обзорный режим в полноценный поисковик по репозиторию.
- Не менять формат существующего proposal workflow, ветвления, PR creation или внешние зависимости.

## Decisions

### Явная rule-based классификация запроса в CLI

Выбор режима должен происходить до создания `proposalrunner.Runner`. Для этого в CLI добавляется простой классификатор с двумя исходами: `project-overview` и `proposal`.

Классификация будет rule-based, по небольшому набору явных информационных фраз о проекте и репозитории, например:

- `расскажи мне о проекте`
- `что это за проект`
- `дай обзор проекта`
- `tell me about the project`
- `what is this project`

Причина выбора:

- это покрывает исходный сценарий без лишней магии;
- правило легко тестировать;
- при сомнении система может безопасно оставаться в режиме `proposal`, чтобы не скрыть реальную задачу на изменение.

Альтернатива: эвристический или LLM-based intent detection. Она избыточна для текущего этапа, требует новых зависимостей и усложняет тестирование.

### Обзор строится из локальных репозиторных источников в фиксированном порядке

Источник ответа должен быть детерминированным. Базовый порядок источников:

1. `README.md` как основной человеко-ориентированный обзор проекта.
2. `docs/proposal-runner.md` как источник деталей текущего workflow.
3. `.env.example` и проверяемые факты из CLI/config-кода как fallback для конфигурационных сведений.

Формат ответа должен быть коротким и структурированным, а не сырым содержимым markdown-файлов. Минимальный ожидаемый состав:

- что такое `orchv3`;
- какой основной workflow поддерживается сейчас;
- какие внешние зависимости и конфигурация нужны;
- где смотреть более подробную документацию.

Альтернатива: жёстко зашитый текст ответа в коде. Она проще в реализации, но быстро начнёт расходиться с документацией.

### Обзорный режим пишет ответ в stdout и не запускает workflow side effects

При выборе `project-overview` CLI печатает итоговый обзор в `stdout` и завершает процесс успешно. Внешние команды, временный каталог, git clone, вызов Codex и GitHub CLI не запускаются.

В `stderr` остаются только структурированные логи самого CLI, если они нужны для наблюдаемости выбранного режима. Это сохраняет совместимость с существующим разделением: результат в `stdout`, служебные сообщения в `stderr`.

Альтернатива: логировать обзор в `stderr`. Это ухудшит пригодность ответа для скриптов и сделает поведение непоследовательным.

### Proposal-конфиг должен грузиться лениво или валидироваться только в proposal-режиме

Сейчас `run(...)` вызывает `config.Load()` до анализа пользовательского ввода, а `ProposalRunnerConfig.Validate()` требует `PROPOSAL_REPOSITORY_URL`. Для обзорного режима это создало бы ложную зависимость от proposal-настроек, хотя никакой proposal workflow не нужен.

Поэтому реализация должна либо:

- разделить базовую загрузку общего CLI-конфига и загрузку proposal-настроек;
- либо читать ввод и выбирать режим до места, где нужна полная proposal-валидация.

Ключевое требование: запрос `Расскажи мне о проекте` должен успешно отработать даже в окружении без `PROPOSAL_REPOSITORY_URL`, `git`, `codex` и `gh`.

Альтернатива: оставлять обязательную загрузку полного конфига до маршрутизации. Это противоречит самой цели локального обзорного режима.

### Реализацию лучше изолировать в маленьком локальном модуле

Сборку обзора стоит вынести из `main.go` в небольшой пакет или helper, отвечающий за:

- определение обзорного запроса;
- чтение локальных файлов;
- построение итогового текста.

Это позволит тестировать логику без запуска всего CLI и без моков внешних команд. При этом выделение должно оставаться минимальным: без дополнительной архитектуры, интерфейсов "на будущее" и сложных абстракций.

Альтернатива: оставить всю логику в `main.go`. Это уменьшит число файлов, но усложнит тесты и размоет ответственность.

## Risks / Trade-offs

- [Слишком узкая классификация не распознает часть обзорных запросов] -> на первом шаге поддерживать только явные фразы и расширять словарь по фактическим кейсам, сохраняя безопасный fallback в proposal-режим.
- [Слишком широкая классификация перехватит реальную задачу на изменение] -> матчить только отчетливо информационные формулировки и не использовать расплывчатые слова вроде `проект` без контекста.
- [Обзор начнет расходиться с текущим репозиторием] -> строить ответ из `README.md` и `docs/proposal-runner.md`, а не из захардкоженного текста.
- [Чтение нескольких файлов сделает вывод хрупким при отсутствии документации] -> предусмотреть fallback-путь и формировать частичный обзор даже если один из источников отсутствует.

## Migration Plan

1. Добавить классификацию входного текста в CLI до вызова proposal runner.
2. Реализовать локальный builder обзорного ответа на базе `README.md` и fallback-источников.
3. Подключить новый режим к `run(...)`, чтобы обзор печатался в `stdout` и завершал выполнение без внешних команд.
4. Добавить unit-тесты для классификации, построения обзора и проверки отсутствия proposal workflow в обзорном режиме.

Откат тривиален: удалить обзорный маршрут и вернуть прежнее ветвление только по признаку пустого/непустого ввода.

## Open Questions

Открытых вопросов нет. Для первой версии достаточно поддержать явные формулировки запроса об обзоре проекта и строить ответ на основе уже существующей документации репозитория.
