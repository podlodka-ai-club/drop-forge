## Context

`orchv3` уже создаёт pull request с OpenSpec proposal и умеет публиковать дополнительный финальный комментарий от имени Codex. При этом ссылка на логи прогона не попадает ни в PR body, ни в итоговый комментарий, хотя для диагностики workflow она полезнее, чем локальный путь к temp-dir.

В проекте уже есть отдельная активная работа по ELK-логированию, но она ещё не заархивирована в базовые spec-и. Поэтому для текущего change важно не завязывать proposal runner на конкретную реализацию Kibana URL, структуру индексов или новый формат полей в логах.

Ограничения:
- решение должно остаться опциональным и не ломать текущий workflow без настройки логов;
- новые runtime-параметры нужно хранить в `.env` и отражать в `.env.example`;
- изменение должно ложиться в существующий поток `config.Load() -> proposalrunner.Run() -> gh pr create`.

## Goals / Non-Goals

**Goals:**
- Дать пользователю готовую ссылку на логи прямо в создаваемом PR.
- Сделать поведение полностью опциональным через конфигурацию.
- Сохранить текущий happy path без изменений, если ссылка на логи не настроена.
- Покрыть поведение тестами на уровне конфигурации и proposal runner.

**Non-Goals:**
- Не строить deep link на конкретный run proposal workflow.
- Не менять формат структурных логов и не добавлять новые поля вроде `run_id`.
- Не добавлять интеграцию с отдельной ticketing-системой вне текущего GitHub PR workflow.
- Не дублировать ссылку и в PR body, и в PR comment.

## Decisions

### 1. Использовать одну опциональную переменную `PROPOSAL_LOGS_URL`

В `internal/config.ProposalRunnerConfig` добавляется поле `LogsURL string`, которое читается из `PROPOSAL_LOGS_URL` без обязательной валидации. Пустое значение трактуется как отключённая функция.

Почему так:
- это минимальное расширение текущей модели конфигурации;
- не нужно знать ничего о backend логов, query-параметрах или формате ссылок;
- `.env.example` остаётся простым и понятным.

Альтернатива:
- `PROPOSAL_LOGS_URL_TEMPLATE` с плейсхолдерами или отдельные параметры для base URL и фильтров.
- Не выбираем это сейчас: шаблоны дают гибкость, но сразу усложняют UX, валидацию и код без подтверждённой необходимости.

### 2. Публиковать ссылку в PR body, а не в отдельном комментарии

Ссылка добавляется в `BuildPRBody(...)` как отдельный блок, например под секцией с описанием задачи. Тогда ссылка всегда видна в основном описании PR и не зависит от успешности шага публикации финального комментария.

Почему так:
- тело PR является главным "тикетом" в текущем workflow;
- не требуется дополнительный вызов `gh pr comment`;
- ссылка становится стабильной частью артефакта ревью.

Альтернативы:
- публиковать ссылку отдельным комментарием;
- добавлять ссылку и в body, и в comment.
- Оба варианта отвергаем: первый делает ссылку менее заметной и хрупко зависит от comment step, второй создаёт дублирование.

### 3. Формировать блок ссылки только при непустом URL

`BuildPRBody` должен получать `taskDescription` и `logsURL` и включать секцию со ссылкой только при непустом, trimmed значении. Если URL отсутствует, текст PR body должен остаться семантически эквивалентным текущему.

Почему так:
- обеспечивает обратную совместимость;
- упрощает тестирование двумя явными сценариями: configured / not configured.

Альтернатива:
- всегда рендерить секцию, например с текстом "Logs not configured".
- Не выбираем: это создаёт шум в PR и не даёт практической пользы.

### 4. Документировать ссылку как операторскую настройку proposal runner

Новая переменная должна быть отражена в `.env.example`, `README.md` и `docs/proposal-runner.md` как опциональная ссылка на страницу логов текущего workflow. Документация должна явно сказать, что ссылка просто пробрасывается в PR body и не валидируется приложением, кроме trim whitespace.

Почему так:
- это соответствует текущему подходу репозитория: все runtime-параметры централизованы в `.env`;
- пользователю сразу ясно, что именно нужно настроить для появления ссылки в PR.

## Risks / Trade-offs

- [Ссылка статическая и не указывает на конкретный запуск] -> принимаем как первый шаг; если появится реальная боль с поиском по общему дашборду, следующим change можно добавить run-specific deep link.
- [Некорректный URL из окружения попадёт в PR body как есть] -> ограничиваемся trim whitespace и документацией; дополнительную URL-валидацию не вводим, чтобы не блокировать нестандартные internal-ссылки.
- [Формат PR body слегка меняется] -> покрываем unit-тестами и сохраняем существующий текст задачи без изменений, добавляя только новый опциональный блок.

## Migration Plan

1. Расширить `ProposalRunnerConfig` и загрузку конфигурации новой опциональной переменной.
2. Изменить построение PR body так, чтобы ссылка добавлялась только при наличии `PROPOSAL_LOGS_URL`.
3. Обновить тесты proposal runner на содержимое `gh pr create --body`.
4. Обновить `.env.example`, `README.md` и `docs/proposal-runner.md`.

Rollback:
- удалить чтение `PROPOSAL_LOGS_URL` из конфига;
- вернуть прежний `BuildPRBody` без блока логов;
- убрать переменную из документации и `.env.example`.

## Open Questions

Нет. В рамках этого change принимается минимальный вариант со статической ссылкой в PR body.
