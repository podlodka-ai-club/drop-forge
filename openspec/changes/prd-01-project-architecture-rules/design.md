## Context

Текущий репозиторий содержит Go-оркестратор и OpenSpec-процесс, но DRO-12 описывает старт нового приложения с отдельными Node.js подпроектами: бекенд на Next.js API routes, фронтенд на React/Vite и PostgreSQL через Docker. Изменение должно создать рабочий локальный контур для PRD-01, не ломая существующий Go-код и обязательные проверки репозитория.

Основные ограничения:

- API развивается contract-first: `docs/openapi.yaml` является источником правды для endpoint-ов.
- Бекенд и фронтенд запускаются независимо через `npm run dev` в своих каталогах.
- Runtime-настройки берутся только из `.env`; `.env.example` хранит только имена переменных без секретов и значений по умолчанию.
- Swagger UI должен быть доступен на `http://localhost:3001/api/docs`.
- PostgreSQL 15 запускается через `docker-compose up` и хранит данные в volume.

## Goals / Non-Goals

**Goals:**

- Создать минимально рабочий каркас `/backend`, `/frontend`, `/docs`, `docker-compose.yml`, `.env.example` и README.
- Настроить Next.js 14 как API-only бекенд с портом `3001`, Prisma и подключением к PostgreSQL.
- Настроить React 18 + Vite + Tailwind CSS фронтенд с портом `3000` и обращением к API через `VITE_API_URL`.
- Описать базовый OpenAPI-контракт, единый response envelope `{ data, error, meta }` и Swagger UI.
- Добавить Prisma-схему всех сущностей из DRO-12, миграцию и seed-данные.
- Документировать команды локального запуска и проверки готовности.

**Non-Goals:**

- Реализация полного набора продуктовых endpoint-ов сверх минимальных служебных endpoint-ов, нужных для проверки каркаса и Swagger UI.
- Реализация пользовательской аутентификации, авторизации, админки или полноценного UI каталога.
- Перенос существующего Go-оркестратора на Node.js или изменение его публичного поведения.
- Настройка production deploy, CI/CD и managed database.

## Decisions

### 1. Node.js подпроекты живут рядом с текущим Go-кодом

`backend/` и `frontend/` создаются как независимые npm-проекты со своими `package.json`, lock-файлами и scripts. Корневые Go-команды остаются валидными.

Альтернатива: переводить весь репозиторий в монорепозиторий с общим package manager workspace. Это преждевременно для PRD-01: задача требует независимого запуска подпроектов, а не общей сборочной системы.

### 2. Next.js используется только как API runtime

Бекенд использует `backend/app/api/**/route.ts`, без страниц и пользовательского Next UI. Для запуска на порту `3001` script `npm run dev` должен явно передавать порт Next.js.

Альтернатива: Express/Fastify. Она проще для API-only сервиса, но противоречит явно заданному стеку DRO-12.

### 3. OpenAPI хранится в корневом `docs/openapi.yaml`

Файл `docs/openapi.yaml` остается общим контрактом между фронтом и беком. Swagger UI в бекенде читает этот файл или его синхронизированную копию без ручного дублирования схем.

Альтернатива: генерировать OpenAPI из кода. Это нарушает процесс "сначала контракт" и делает реализацию источником правды.

### 4. Единый response envelope применяется ко всем API endpoint-ам

Бекенд должен отдавать ответы вида `{ data, error, meta }`. Для успешных ответов `error` равен `null`, для ошибок `data` равен `null`, а `meta` содержит только переносимые служебные поля при необходимости.

Альтернатива: отдавать разные формы для разных endpoint-ов. Это ускоряет первые ручки, но сразу ломает контракт с фронтендом.

### 5. Prisma-схема отражает домен без преждевременных сервисных абстракций

Сущности DRO-12 моделируются в `backend/prisma/schema.prisma` с явными связями, timestamps там, где они указаны, и enum для статуса лида. Seed лежит в `backend/prisma/seed` или `backend/prisma/seed.ts` и создает репрезентативные данные.

Альтернатива: начать с SQL-файлов без Prisma. Это уменьшает зависимости, но противоречит заданному ORM-стеку и усложняет `npx prisma studio`.

### 6. Конфигурация разделена по runtime-потребителям

Корневой `.env.example` содержит ключи для Docker, бекенда и фронтенда без значений. Реальные значения разработчик кладет в `.env`, а подпроекты загружают нужные переменные стандартными средствами Next/Vite/Prisma.

Альтернатива: держать отдельные `.env.example` в `backend/` и `frontend/`. Это может понадобиться позже, но для начального каркаса единый корневой шаблон проще и лучше соответствует текущим правилам репозитория.

## Risks / Trade-offs

- [Риск] В одном репозитории появляются Go и Node.js toolchains → [Mitigation] README явно разделяет команды Go-проекта, бекенда, фронтенда и Docker; существующие Go-проверки остаются обязательными.
- [Риск] `.env.example` без значений хуже подходит для copy-paste запуска → [Mitigation] README описывает назначение переменных и пример локальных значений текстом, не нарушая правило шаблона без дефолтов.
- [Риск] Swagger UI может разойтись с `docs/openapi.yaml`, если файл копировать вручную → [Mitigation] бекенд читает корневой контракт напрямую либо копирование автоматизируется npm-script-ом.
- [Риск] Seed на 50-100 программ может стать шумным для поддержки → [Mitigation] seed должен быть детерминированным, компактным и построенным из небольших массивов стран, городов, направлений и университетов.
- [Риск] `Favorite` допускает nullable `university_id` и `program_id`, что может создать пустые записи → [Mitigation] Prisma-схема и API-валидация должны требовать ровно одну цель избранного на уровне приложения; DB constraint можно добавить позже отдельной миграцией, если потребуется.

## Migration Plan

1. Добавить файловую структуру, package manifests и базовую конфигурацию бекенда, фронтенда, Docker и окружения.
2. Добавить `docs/openapi.yaml` с базовыми схемами envelope, ошибками и служебными endpoint-ами.
3. Добавить Prisma schema, миграцию и seed.
4. Добавить Swagger UI route и минимальные API routes для проверки запуска.
5. Добавить минимальный Vite/Tailwind фронтенд, который читает `VITE_API_URL` и может проверить доступность API.
6. Обновить README командами запуска и готовности.
7. Выполнить форматирование и тесты: `go fmt ./...`, `go test ./...`, а также доступные npm-проверки в `backend` и `frontend`.

Rollback: так как это начальный каркас без миграции production-данных, откат выполняется удалением добавленных каталогов и файлов change-а. Docker volume с локальными данными можно удалить отдельной командой, если разработчику нужно полностью очистить окружение.

## Open Questions

- Нужен ли root-level npm workspace для общей установки зависимостей?
  - Почему важно: workspace может упростить команды из корня, но добавит дополнительную инфраструктуру сверх требований DRO-12.
  - Рекомендация: не добавлять workspace в рамках PRD-01, оставить независимый запуск из `backend/` и `frontend/`.
- Какие минимальные endpoint-ы включать в первый OpenAPI-контракт помимо Swagger UI?
  - Почему важно: задача задает правила и модель, но не перечисляет публичные продуктовые endpoint-ы.
  - Рекомендация: добавить только служебный `/api/health` и схемы доменных моделей; продуктовые endpoint-ы заводить отдельными contract-first изменениями.
