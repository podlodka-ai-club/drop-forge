## Context

Сейчас proposal orchestration строит `ProposalInput` из Linear-задачи и уже поддерживает отсутствие описания или комментариев как валидный вход. DRO-49 проверяет самый короткий реальный сценарий: задача имеет стабильную идентичность (`DRO-49`) и заголовок (`Просто тест`), но не несет продуктового описания.

Главная опасность такого входа не в runtime-ошибке, а в неявной интерпретации: агент может попытаться превратить пустую задачу в произвольную функциональность. Поэтому контракт должен требовать явного smoke/validation framing в prompt и тестов на этот случай.

## Goals / Non-Goals

**Goals:**

- Сохранить успешную обработку Linear-задач без описания и комментариев.
- Сделать низкоконтекстный вход явным для proposal runner: источник, отсутствие описания, отсутствие комментариев, ожидаемый формат допущений.
- Добавить regression test для задачи с идентификатором, заголовком и пустыми description/comments.

**Non-Goals:**

- Не добавлять отдельный workflow state для тестовых задач.
- Не менять Linear API, GitHub workflow, CLI contract или OpenSpec schema.
- Не реализовывать специальную бизнес-фичу по заголовку `Просто тест`.

## Decisions

1. **Использовать существующий proposal route.**

   Низкоконтекстная задача остается обычной ready-to-propose задачей. Это проверяет реальный путь оркестрации и не создает параллельный тестовый режим, который мог бы расходиться с production flow.

2. **Уточнить только формирование agent prompt.**

   Prompt должен явно содержать, что описание не предоставлено, комментарии отсутствуют, а proposal должен фиксировать принятые допущения. Альтернатива с добавлением поля `IsSmokeTest` в `ProposalInput` отклонена: текущий контракт уже может выразить это в текстовом контексте без расширения модели данных.

3. **Покрыть поведение unit-тестом с fake-зависимостями.**

   Тест должен проверять подготовленный `ProposalInput`, а не вызывать Codex CLI или Linear. Это соответствует существующему подходу к тестируемым orchestration dependencies и сохраняет быстрый `go test ./...`.

## Risks / Trade-offs

- Низкоконтекстная задача все равно может привести к proposal с открытыми вопросами, а не к полноценной реализации -> это ожидаемо; smoke-сценарий проверяет устойчивость процесса, а не продуктовую полноту.
- Prompt-based framing слабее, чем отдельный структурный флаг -> выбран этот вариант, потому что он не расширяет публичный контракт ради одного validation-сценария.
- Если в будущем появятся формальные типы Linear-задач, этот сценарий может быть перенесен на структурное поле -> текущий дизайн не блокирует такой перенос.
