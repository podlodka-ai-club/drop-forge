## Context

В репозитории уже есть несколько полезных, но разноуровневых источников контекста:

- `README.md` дает быстрый вход в проект и инструкцию по запуску;
- `architecture.md` описывает целевую архитектурную рамку и границы ответственности;
- `docs/proposal-runner.md` и `docs/linear-task-manager.md` раскрывают отдельные подсистемы;
- `openspec/specs/*` фиксируют требования по capability.

Для читателя, который хочет быстро понять "что это за проект и как он устроен сейчас", между README и архитектурным документом остается зазор. README слишком краток для содержательного обзора, а `architecture.md` уже уходит в модель акторов и будущую рамку системы. Изменение остается документационным: runtime-поведение, конфигурация и orchestration flow не меняются.

## Goals / Non-Goals

**Goals:**

- Добавить один обзорный документ, который объясняет назначение `orchv3`, текущий proposal-stage scope и ключевые компоненты без погружения в код.
- Показать читателю маршрут по документации: от README к обзорному документу, затем к архитектуре и subsystem-specific docs.
- Синхронизировать обзор с текущей реализацией и не описывать несуществующие режимы или будущие возможности как уже готовые.
- Сохранить существующую роль README как короткой entrypoint-страницы.

**Non-Goals:**

- Не менять CLI, `.env`, OpenSpec workflow, Linear integration или proposal runner.
- Не переписывать `architecture.md` в упрощенный формат и не дублировать его полностью в новом документе.
- Не добавлять новую иерархию документации за пределами одного обзорного файла и минимального обновления README.

## Decisions

### Добавить отдельный `docs/project-overview.md`

Новый обзорный документ будет стоять между корневым README и подробной технической документацией. Он должен отвечать на вопросы:

- какую задачу решает `orchv3`;
- что умеет текущая версия;
- какие основные внутренние роли участвуют в proposal-stage;
- какие документы читать дальше для деталей.

Альтернатива: расширить `README.md` до полноценного обзора. Это ухудшит его как entrypoint-документ и смешает быстрый старт с обзором архитектурного уровня.

### Использовать существующие документы как источники правды, а не создавать новую трактовку проекта

Содержание обзорного документа должно собираться из уже существующих артефактов: `README.md`, `architecture.md`, `docs/proposal-runner.md`, `docs/linear-task-manager.md` и актуальных OpenSpec specs. Новый документ не должен вводить собственные термины или обещания, которые расходятся с этими файлами.

Альтернатива: написать "маркетинговый" обзор без жесткой привязки к текущему состоянию кода. Это быстрее, но создаст расхождение между документацией и фактической реализацией.

### README должен явно указывать на следующий шаг чтения

После изменения README останется коротким, но в нем появится явная ссылка на обзорный документ проекта. Это сохранит навигационную роль README и снимет проблему, когда читателю приходится самостоятельно выбирать между `architecture.md` и несколькими файлами в `docs/`.

Альтернатива: оставить навигацию неявной и рассчитывать, что читатель найдет нужные документы сам. Это не решает исходную задачу "рассказать о проекте" быстрым и понятным способом.

## Risks / Trade-offs

- [Обзорный документ начнет дублировать README и architecture.md] -> сделать его связующим, а не исчерпывающим: краткий обзор плюс ссылки на детальные документы.
- [Документ устареет при изменении workflow] -> опираться только на уже принятые capabilities и текущий код, не документировать гипотетические будущие сценарии.
- [README потеряет фокус быстрого старта] -> ограничить изменение README навигационной ссылкой и коротким пояснением, без разрастания разделов.
- [Появится неоднозначность между "текущим состоянием" и "целевой архитектурой"] -> в обзорном документе явно отмечать, что `architecture.md` описывает архитектурную рамку, а обзорный документ фиксирует текущее понимание проекта на уровне продукта и потока работ.
