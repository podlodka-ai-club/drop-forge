## Context

Сейчас проект реализует только один orchestration flow: CLI принимает текст задачи и передает его в `proposalrunner`, который клонирует внешний репозиторий, запускает Codex, создает commit, push, pull request и PR comment. Источником задачи пока всегда выступает человек или внешний скрипт, а слоя управления внешней очередью задач в проекте нет.

В целевой архитектуре между Linear и будущим `CoreOrch` нужен отдельный `TaskManager`. Он не оркестрирует выполнение задачи сам, а отвечает за чтение и запись данных в Linear: по запросу `CoreOrch` выбрать задачи из конкретного проекта и нужных state'ов, отдать наружу описание и комментарии задачи, а затем применить обратные изменения вроде state transition, комментария и PR link. Это особенно важно для HITL-сценария: после reject человек оставляет комментарии в Linear, и `CoreOrch` должен через тот же `TaskManager` снова получить обновленную задачу вместе с feedback для следующего запуска executor.

Ограничения проекта:

- держать первую версию простой и расширяемой, без преждевременных абстракций;
- все runtime-параметры хранить в `.env` и поддерживать в `.env.example`;
- новые сценарии должны быть тестируемыми без реальных сетевых вызовов;
- structured logging уже является подтвержденным контрактом проекта и должен сохраняться для нового orchestration-layer.
- `CoreOrch` как отдельный компонент в коде пока отсутствует, поэтому текущий change должен подготовить именно интеграционный слой, а не весь orchestration loop.

## Goals / Non-Goals

**Goals:**

- Добавить отдельный `TaskManager`, который работает с Linear как с внешним источником данных для будущего `CoreOrch`.
- Поддержать выбор задач только из одного настроенного проекта Linear и только из конфигурируемых state ID маршрутов `ready to propose`, `ready to code` и `ready to archive`.
- Возвращать задачу в форме, пригодной для `CoreOrch`: идентификатор, ключ, описание, текущий state и комментарии к задаче, чтобы оркестратор мог передать этот контекст нужному executor.
- Поддержать операции обратной записи в Linear: перевод задачи в другой state, добавление комментария и привязка PR.
- Сделать новый слой тестируемым через подменяемый Linear client и без зависимости от реальных executor'ов.

**Non-Goals:**

- Не реализовывать в этом change `CoreOrch`, polling loop и dispatch задач в executor'ы.
- Не превращать `TaskManager` в универсальный orchestration-слой.
- Не менять существующий контракт `codex-proposal-pr-runner`, если для интеграции достаточно подготовить данные задачи для будущего executor.
- Не вводить отдельное хранилище состояния вне Linear в первой версии.

## Decisions

### `TaskManager` остается Linear-facing слоем, а не `CoreOrch`

`TaskManager` нужно выделить в отдельный пакет, но его ответственность должна совпадать с целевой архитектурой:

- запросить список задач из Linear по настроенному проекту и state ID;
- вернуть наружу данные задачи, включая описание и комментарии;
- применить операции записи обратно в Linear: move, comment, add PR.

При этом решение, в какой именно target state переводить задачу после `propose/code/archive`, остается у `CoreOrch`. `TaskManager` не выбирает review-этап сам, а только предоставляет операцию `MoveTask` и доступ к нужным target state ID через конфиг.

Альтернатива - встроить orchestration прямо в `TaskManager`. Это упростит первый коммит, но смешает Linear-интеграцию и логику `CoreOrch`, которую вы отдельно выделили в целевой схеме.

### Интеграция с Linear через project-scoped выборку и внутреннюю доменную модель

В первой версии нужен отдельный адаптер `internal/linear`, который инкапсулирует доступ к Linear и отдает `TaskManager` минимальную внутреннюю модель задачи: идентификатор, ключ, заголовок, описание, текущий state и метаданные, нужные для комментария/перехода/PR link.

Выборка задач должна быть ограничена одним настроенным проектом Linear. Фильтр по project должен применяться вместе с фильтром по state ID, чтобы `CoreOrch` вообще не видел задачи из чужих проектов.

`TaskManager` не должен зависеть от сырых транспортных структур Linear API. Это позволяет:

- подменять client в unit-тестах;
- не разносить детали внешнего API по orchestration-коду;
- позже заменить транспортный слой без переписывания бизнес-логики менеджера.

Альтернатива - использовать структуры API прямо в `TaskManager`. Это быстрее в короткую, но сильнее связывает orchestration со схемой Linear.

### Комментарии задачи входят в контракт чтения задачи

Для proposal HITL-сценария недостаточно только описания issue. `TaskManager` должен возвращать вместе с задачей список комментариев или эквивалентную историю human feedback, чтобы `CoreOrch` мог повторно вызвать proposal workflow с замечаниями после reject.

В первой итерации `TaskManager` возвращает все доступные комментарии задачи без дополнительной фильтрации по автору или релевантности. Отбор нужного feedback остается ответственностью `CoreOrch` или верхнего orchestration-слоя.

Минимальный контракт чтения задачи должен включать:

- идентификатор задачи;
- issue key;
- заголовок;
- описание;
- текущий state;
- комментарии;
- при наличии - связанные metadata, нужные для PR synchronization.

Альтернатива - читать комментарии отдельным вызовом только в proposal-сценарии. Это делает контракт менее прозрачным и перекладывает знание о Linear data model в `CoreOrch`.

### Операции записи обратно в Linear должны быть отдельными методами `TaskManager`

С учетом целевой схемы `TaskManager` должен явно поддерживать отдельные операции записи:

- `MoveTask` или эквивалентный перевод задачи в новый state;
- `AddComment`;
- `AddPR`.

Такой интерфейс лучше совпадает с последовательностью вызовов из вашей диаграммы: `CoreOrch` решает, когда и что обновить, а `TaskManager` только выполняет операцию в Linear.

В первой итерации поддержка PR association ограничивается записью через `AddPR`. Чтение уже существующих PR association из Linear не является обязательной частью read-модели и может быть добавлено позже при появлении явного сценария в `CoreOrch`.

Для review-этапов в конфиге должны быть отдельные target state ID:

- `Need Proposal Review`
- `Need Code Review`
- `Need Archive Review`

Это позволит `CoreOrch` после успешного завершения executor'а явно вызывать `MoveTask` в правильный review-state, не захардкодивая эти идентификаторы внутри orchestration-кода.

### Конфигурация state mapping и доступа к Linear через `.env`

Настройки Linear нужно держать централизованно в `internal/config`, рядом с уже существующим proposal-конфигом. Для первой версии достаточно:

- параметров подключения к Linear;
- идентификатора или имени одного целевого проекта Linear;
- идентификаторов state для `ready to propose`, `ready to code`, `ready to archive`;
- идентификаторов state для `need proposal review`, `need code review`, `need archive review`;
- параметров, нужных для операций изменения задачи и привязки PR.

`.env.example` должен содержать все новые ключи без значений. Это соответствует текущему контракту проекта и упрощает локальную настройку.

### Structured logging сохранить на уровне `taskmanager` и `linear`

Новый flow должен логировать шаги так же, как текущий proposal workflow. Минимальный стабильный набор модулей для первой версии:

- `taskmanager` - решения уровня интеграционного слоя;
- `linear` - чтение задач и запись изменений обратно.

Это сохраняет единый машинно-читаемый поток логов и делает поведение Linear-интеграции наблюдаемым до появления `CoreOrch`.

## Risks / Trade-offs

- [Маршрутизация по human-readable именам state может быть хрупкой] -> использовать в конфиге устойчивые идентификаторы state и изолировать mapping в одном месте.
- [Фильтр только по state без project приведет к лишним задачам в orchestration] -> сделать project filter обязательной частью конфига и валидации.
- [Комментарии могут быть нужны не всем сценариям, но станут частью базового контракта] -> принять более богатую модель чтения задачи, чтобы не раскалывать интерфейс позже из-за HITL.
- [Интеграция с внешним API усложняет тесты] -> все сетевые вызовы прятать за интерфейсом Linear client и покрывать `TaskManager` unit-тестами на fake-клиенте.
- [В коде еще нет `CoreOrch`] -> зафиксировать границу ответственности в design/spec сейчас, чтобы не размыть архитектуру при первом implementation change.

## Migration Plan

1. Добавить новую capability и runtime-конфиг для Linear, включая project filter.
2. Реализовать тонкий Linear client и `TaskManager` как слой чтения/записи данных задачи.
3. Поддержать возврат описания и комментариев задачи в едином контракте чтения.
4. Добавить методы записи обратно в Linear: move, comment, add PR.
5. Покрыть manager и config unit-тестами, затем обновить документацию.
