## Context

`TaskManager` уже возвращает описание и комментарии Linear в `taskmanager.Task`. `coreorch` строит входные структуры для proposal, apply и archive через общий agent prompt, а `applyrunner` и `archiverunner` передают этот prompt в Codex CLI как `Task context`.

Текущий риск в том, что контракт для `apply` и `archive` сформулирован слишком общо: спека говорит о контексте задачи, но не требует проверять, что человеческие комментарии действительно присутствуют в prompt этих стадий. Для DRO-51 важно сделать это явным, потому что комментарии часто содержат исправления после ревью proposal/code/archive.

## Goals / Non-Goals

**Goals:**

- Гарантировать, что `ApplyInput.AgentPrompt` и `ArchiveInput.AgentPrompt` содержат блок комментариев Linear.
- Сохранить единый формат task prompt для proposal, apply и archive, чтобы не расходились правила форматирования описания, комментариев и пустых значений.
- Покрыть поведение unit-тестами на уровне `coreorch` и, при необходимости, runner prompt tests.

**Non-Goals:**

- Не менять GraphQL-загрузку комментариев из Linear: она уже относится к `linear-task-manager`.
- Не добавлять новые параметры `.env`.
- Не менять state machine стадий, порядок переходов и работу с PR/branch source.

## Decisions

- Использовать общий builder task prompt в `coreorch` для `apply` и `archive`.
  - Причина: комментарии уже нормализуются там же, где собираются ID, identifier, title и description.
  - Альтернатива: добавлять отдельные поля `Comments` в `ApplyInput`/`ArchiveInput`. Это усложняет контракты runner-пакетов без пользы, потому что Codex CLI все равно получает один текстовый prompt.

- Явно проверять блок `Comments:` в тестах `BuildApplyInput` и `BuildArchiveInput`.
  - Причина: это ближайшая граница, где задача Linear превращается во вход стадии.
  - Альтернатива: проверять только финальный stdin Codex executor. Такой тест полезен, но он не доказывает, что orchestration не потеряло комментарии до runner.

- Представлять отсутствие комментариев строкой `No comments available.`.
  - Причина: это уже используется для proposal prompt и дает агенту явный сигнал, что комментарии были проверены.
  - Альтернатива: оставлять блок пустым. Это хуже для диагностики и может выглядеть как ошибка форматирования prompt.

## Risks / Trade-offs

- Изменение формата prompt может повлиять на стабильность тестов и downstream-ожидания -> сохранить существующий формат блока комментариев и добавить только недостающие проверки.
- Очень длинные обсуждения Linear увеличат размер prompt -> в рамках этого change не вводим лимиты или суммаризацию, потому что требование DRO-51 просит подтягивать комментарии полностью как контекст выполнения.
- Комментарии могут содержать пустые тела или отсутствующих авторов -> использовать существующие fallback-значения `Unknown author` и `(empty comment)`.

## Migration Plan

Миграция данных не требуется. После реализации достаточно запустить `go fmt ./...` и `go test ./...`; новые проверки должны проходить без изменения runtime-конфигурации.
