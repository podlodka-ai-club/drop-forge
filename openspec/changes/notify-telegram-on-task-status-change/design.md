## Context

`internal/taskmanager.Manager` сейчас выполняет операции с Linear напрямую: читает задачи, двигает задачу в новый workflow state, добавляет комментарий и PR. Смена статуса происходит через `MoveTask(ctx, taskID, stateID)` и после успешной Linear mutation только логируется.

DRO-23 требует уведомлять Telegram о смене статуса задачи и явно просит Event Driven Design. Значит реакция на смену статуса должна быть подписчиком на доменное событие, а не новой прямой зависимостью вызывающего orchestration-кода от Telegram.

## Goals / Non-Goals

**Goals:**

- Публиковать событие `TaskStatusChanged` после успешного изменения статуса задачи в Linear.
- Добавить простой синхронный in-process dispatcher, который позволяет регистрировать несколько обработчиков события.
- Реализовать Telegram-уведомитель как обработчик события, а не как код внутри orchestration caller.
- Хранить Telegram runtime-настройки в централизованной конфигурации и `.env.example`.
- Сохранить текущий сценарий вызова `MoveTask`: caller не должен вызывать Telegram напрямую.

**Non-Goals:**

- Не вводить брокер сообщений, фоновые worker-ы, durable queue или retry-планировщик.
- Не реализовывать двустороннюю Telegram-интеграцию, команды бота или webhook-и.
- Не менять модель workflow states в Linear сверх данных, нужных для события и уведомления.
- Не гарантировать доставку Telegram-уведомления после завершения процесса, если процесс был остановлен между Linear mutation и обработкой события.

## Decisions

### Событие публикуется в `Manager.MoveTask` после успешной Linear mutation

`MoveTask` остается единственной точкой записи статуса через task manager. После `client().MoveTask(ctx, taskID, stateID)` manager создает `TaskStatusChangedEvent` и передает его dispatcher-у. Если Linear возвращает ошибку или `success=false`, событие не публикуется.

Альтернатива: публиковать событие из вызывающего orchestration-кода после `MoveTask`. Это хуже для DRO-23, потому что каждый caller должен знать о событии и порядок реакций снова будет размазан по вызывающему коду.

### Dispatcher синхронный и in-process

На текущем этапе достаточно интерфейса вида `EventPublisher`/`EventHandler`, где manager вызывает `Publish(ctx, event)`, а dispatcher последовательно вызывает зарегистрированные handler-ы. Ошибка handler-а возвращается caller-у с контекстом, чтобы не скрывать сбой реакции.

Альтернатива: асинхронная очередь с retry. Это преждевременное усложнение для одного notification subscriber-а и текущей ранней стадии проекта.

### Telegram реализуется через стандартный `net/http`

Telegram Bot API для `sendMessage` можно вызвать обычным `POST` запросом с JSON body. Внешний SDK не нужен: он добавит зависимость ради одного endpoint-а и усложнит тестирование.

Альтернатива: официальный или сторонний Telegram SDK. Его можно рассмотреть позже, если появятся команды, webhook-и, media attachments или расширенная обработка ошибок API.

### Событие содержит task ID, target state ID и best-effort имя статуса

Текущий публичный `MoveTask` получает только `taskID` и `stateID`, поэтому событие не должно требовать предварительного повторного чтения задачи из Linear. Для человекочитаемого сообщения manager может сопоставить `stateID` с известными state IDs из `LinearTaskManagerConfig`; если имени нет, уведомление использует state ID.

Альтернатива: перед сменой статуса получать полную задачу и target state из Linear. Это дороже, добавляет сетевые вызовы и не нужно для минимального рабочего уведомления.

### Telegram-конфиг отделяется от Linear-конфига

В `config.Config` добавляется `Telegram TelegramConfig` с `BotToken` и `ChatID`. Валидация Telegram-конфига выполняется при построении уведомителя или когда Telegram-уведомления включены. Это не должно ломать существующие task manager тесты и сценарии без Telegram.

Альтернатива: включить Telegram поля в `LinearTaskManagerConfig`. Это смешает внешний канал уведомления с настройками Linear и ухудшит границы ответственности.

## Risks / Trade-offs

- [Risk] Linear status уже изменен, а Telegram отправка завершилась ошибкой → возвращать ошибку dispatcher-а с контекстом и логировать ее; автоматический rollback статуса не выполнять.
- [Risk] Синхронная отправка Telegram увеличивает latency `MoveTask` → принять как простой начальный вариант; при необходимости позже заменить handler на асинхронный без изменения caller-кода.
- [Risk] Событие не содержит полного старого и нового статуса → для DRO-23 достаточно факта смены и целевого статуса; старый статус можно добавить позже, когда появится реальная потребность.
- [Risk] Telegram token является секретом → читать только из окружения/`.env`, не задавать значения по умолчанию и не коммитить `.env`.

## Migration Plan

1. Добавить типы событий и dispatcher в зоне task manager или небольшом внутреннем пакете, сохранив простые интерфейсы для тестов.
2. Подключить публикацию события после успешного `MoveTask`.
3. Добавить Telegram config, notifier и HTTP-клиент с подменяемым transport/doer для unit-тестов.
4. Собрать зависимости в CLI entrypoint: если Telegram config заполнен, зарегистрировать Telegram handler в dispatcher-е task manager.
5. Обновить `.env.example`, `architecture.md`, тесты и документацию публичного поведения по мере необходимости.
6. Rollback: отключить Telegram-конфиг или не регистрировать Telegram handler; Linear state transition продолжит работать без уведомлений.
