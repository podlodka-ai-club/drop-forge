## Context

Сейчас быстрый вход в проект разбит между `README.md`, `architecture.md` и специализированными документами в `docs/`. Такой набор полезен для детального чтения, но не решает задачу короткого объяснения проекта для человека, который видит репозиторий впервые или получает задачу уровня "расскажи о проекте".

Изменение затрагивает только документационный слой, но оно кросс-файловое: нужно согласованно развести роли `README.md`, нового обзорного документа и уже существующих deep-dive материалов. При этом важно не дублировать архитектурный документ и не вводить новый источник истины для runtime-поведения.

## Goals / Non-Goals

**Goals:**

- Ввести один короткий документ, который отвечает на вопросы "что это за проект", "что уже реализовано" и "куда читать дальше".
- Сделать `README.md` точкой входа, которая быстро направляет читателя в обзор проекта и в более глубокие документы.
- Явно развести текущую реализованную вертикаль и целевую архитектуру, чтобы документация не обещала лишнего.
- Зафиксировать это поведение в OpenSpec-требованиях к документации.

**Non-Goals:**

- Не менять CLI, orchestration flow, runtime-конфигурацию или структуру Go-пакетов.
- Не переписывать `architecture.md` в формат onboarding-документа.
- Не дублировать в новом обзоре полные инструкции запуска, настройки окружения или подробности Linear/proposal-runner workflow.

## Decisions

### Выделить отдельный `docs/project-overview.md`

Короткий обзор лучше держать отдельным документом, а не раздувать `README.md` или `architecture.md`.

- `README.md` остается компактным entrypoint на уровне репозитория.
- `architecture.md` остается документом про границы ответственности и целевую схему.
- `docs/project-overview.md` берет на себя onboarding-роль: назначение проекта, текущий scope, ключевые компоненты, ссылки дальше.

Альтернатива - расширить только `README.md`. Это упростит структуру, но снова смешает краткий обзор, запуск и подробности по workflow в одном файле.

### Сделать README маршрутизатором, а не дубликатом обзора

`README.md` должен коротко представить проект и сразу направить читателя в `docs/project-overview.md`, `docs/proposal-runner.md`, `docs/linear-task-manager.md` и `architecture.md` по задачам чтения.

Альтернатива - полностью перенести обзор в README и оставить остальные документы как secondary references. Это увеличивает риск рассинхронизации, потому что README придется поддерживать как самый подробный документ сразу по нескольким темам.

### Зафиксировать различие между "текущим состоянием" и "целевой архитектурой"

В обзоре нужно явно разделить:

- что уже работает сейчас в коде;
- какие компоненты являются целевой архитектурной рамкой;
- какие документы описывают детали каждого слоя.

Альтернатива - описывать проект только в терминах целевой архитектуры. Это проще написать, но вводит читателя в заблуждение относительно реально реализованного scope.

## Risks / Trade-offs

- [Обзорный документ начнет дублировать README и architecture.md] → Жестко ограничить его роль onboarding-контекстом и ссылками на углубленные документы.
- [Документация устареет после следующих архитектурных изменений] → Зафиксировать требование, что обзор должен отражать только текущее поведение и актуальные границы системы.
- [Появится слишком много точек входа в документацию] → Сделать README единственным маршрутизатором, а остальные документы специализированными.

## Migration Plan

1. Добавить новую capability `project-overview-doc` и delta для `project-readme`.
2. Реализовать `docs/project-overview.md` с кратким описанием проекта и ссылками на существующие документы.
3. Обновить `README.md`, чтобы он ссылался на обзор как на основной быстрый вход.
4. Проверить, что обзор не противоречит `architecture.md` и текущему README.

## Open Questions
