## Context

Текущий проект минимален: `cmd/orchv3` загружает конфигурацию из environment через `internal/config` и пишет стартовый лог. Новая возможность затрагивает несколько зон: загрузку `.env`, orchestration flow, запуск внешних CLI и публикацию PR в удаленном GitHub-репозитории.

Основной пользовательский сценарий: оператор передает текстовое описание задачи; приложение клонирует репозиторий из ENV во временную директорию, запускает `codex` со skill `openspec-propose`, затем создает PR с полученными OpenSpec-изменениями и возвращает URL.

Ограничения проекта:

- основной язык - Go;
- runtime-параметры должны жить в `.env`;
- шаблоны переменных должны быть без секретов и значений;
- внешние инструменты допустимы только при понятной необходимости;
- все шаги и взаимодействие с Codex CLI должны быть видны в консоли.

## Goals / Non-Goals

**Goals:**

- Дать модуль с функцией, принимающей `context.Context` и строку описания задачи, возвращающей ссылку на PR или ошибку с контекстом.
- Централизованно загрузить настройки из `.env` и environment.
- Создавать уникальную временную директорию и клонировать туда GitHub-проект через `git clone`.
- Запускать Codex CLI в директории клона со skill `openspec-propose` и исходным описанием задачи.
- Потоково логировать prompt, stdout/stderr Codex CLI, stderr/stdout git и шаги создания PR.
- Создавать ветку, коммитить OpenSpec-артефакты, пушить ветку и открывать pull request.
- Сделать логику тестируемой без реального GitHub и без запуска настоящего Codex CLI.

**Non-Goals:**

- Не реализовывать собственный Git-клиент.
- Не реализовывать полноценный GitHub API-клиент, пока сценарий можно надежно закрыть `gh`.
- Не запускать реализацию задач из OpenSpec; модуль отвечает только за proposal/spec/design/tasks и PR.
- Не управлять параллельным пулом задач и очередями.
- Не хранить секреты или значения runtime-настроек в репозитории.

## Decisions

### Модуль orchestration в отдельном пакете

Добавить пакет `internal/proposalrunner` с типом `Runner` и зависимостями через интерфейсы:

- `CommandRunner` для запуска внешних команд;
- `Logger` или `io.Writer` для консольных логов;
- `TempDirProvider` или использование `os.MkdirTemp` внутри при тестах через `t.TempDir`;
- структура `Config` с URL репозитория, base branch, именем удаленного, префиксом ветки и путями к CLI.

Альтернатива - держать все в `cmd/orchv3/main.go`. Это быстрее на старте, но затруднит тесты, подмену команд и повторное использование модуля. Отдельный пакет лучше соответствует требованию "разработать модуль".

### `.env` читать в конфигурационном пакете без внешней зависимости

Расширить `internal/config.Load` чтением `.env`: простой parser формата `KEY=VALUE`, пропуск пустых строк и комментариев, без shell expansion. Значения из уже установленного environment должны иметь приоритет над `.env`, чтобы CI и оператор могли переопределять параметры.

Альтернатива - добавить `godotenv`. Для текущего формата стандартной библиотеки достаточно, а правила проекта требуют сначала обосновывать внешние инструменты.

### Внешние CLI запускать через `os/exec`

Использовать реальные CLI `git`, `codex` и `gh`, потому что:

- `git clone`, commit и push надежнее поручить Git CLI;
- Codex CLI является требуемым внешним исполнителем;
- `gh pr create` закрывает создание PR без собственного GitHub API-клиента и лишней обработки токенов.

Пути к бинарям сделать настраиваемыми через ENV, чтобы тестовые и production-окружения могли подменять команды.

Для первой версии создание PR остается через авторизованный `gh` CLI. Собственный GitHub API-клиент на стандартной библиотеке не нужен, пока `gh` покрывает сценарий и упрощает работу с аутентификацией.

### Логирование сделать потоковым и структурированным по шагам

Каждый шаг пишет понятный префикс в консоль:

- `temp`: создание директории;
- `git`: clone, checkout branch, status, add, commit, push;
- `codex`: command, prompt, stdout/stderr;
- `github`: создание PR и URL.

Для Codex CLI важно логировать и prompt до запуска, и весь stdout/stderr без буферизации до конца процесса. Если Codex CLI пишет рассуждения агента в stderr/stdout, приложение не фильтрует их и выводит в консоль как есть.

### Формат запуска Codex CLI

Команда должна использовать текущий локальный non-interactive формат Codex CLI, без настраиваемого шаблона argv через ENV. Prompt должен включать явное указание использовать skill `openspec-propose` и пользовательское описание задачи.

Начальный вариант:

- argv: `codex exec --cd <clone-dir> -`;
- prompt передается через stdin;
- prompt содержит markdown-ссылку на skill или имя skill, если путь внутри целевого проекта недоступен;
- пользовательское описание передается как часть prompt, а не через shell interpolation.

Builder-метод все равно нужен, но только чтобы централизованно собрать текущий локальный формат команды и покрыть его тестом. В первой версии не добавлять ENV-переменную для произвольного argv-шаблона.

### Ветка и PR

После успешного Codex CLI:

- вычислить slug из описания задачи или использовать настраиваемый префикс;
- создать/переключить рабочую ветку;
- проверить наличие изменений;
- добавить только OpenSpec-изменения и связанные файлы, которые создал Codex;
- сделать коммит;
- выполнить push;
- создать PR через `gh pr create --json url --jq .url`.
- если есть открытые вопросы к реализации, добавить их отдельным комментарием в созданный PR и залогировать URL или результат команды.

Если изменений нет, вернуть ошибку с контекстом, чтобы оператор видел, что Codex не создал артефакты.

### Temp-директория

В первой версии временная директория сохраняется по умолчанию после завершения workflow, чтобы оператор мог диагностировать вывод Codex и состояние клонированного проекта. Для автоматизированных запусков добавить ENV-флаг, который включает удаление temp-директории после успешного или неуспешного завершения.

## Risks / Trade-offs

- [Разные версии Codex CLI могут иметь разные флаги] -> изолировать построение команды, покрыть тестом ожидаемый argv и документировать поддерживаемый формат.
- [Codex CLI может требовать интерактивности] -> запускать в non-interactive режиме, если CLI поддерживает флаг; иначе явно логировать зависший/ошибочный запуск и поддержать timeout через context.
- [Создание PR через `gh` требует авторизации] -> валидировать наличие `gh`, логировать ошибку с подсказкой по отсутствующей авторизации, не хранить токены в коде.
- [В `.env` могут попасть секреты] -> `env.dist` и `.env.example` содержат только ключи без значений; `.env` остается локальным.
- [Временные директории могут занимать место] -> в первой версии сохранять temp-dir по умолчанию ради диагностики, логировать путь и добавить ENV-флаг для включения cleanup.
- [Codex может изменить неожиданные файлы] -> перед коммитом логировать `git status --short`; начально коммитить весь diff целевого клона, потому что Codex/OpenSpec может создавать несколько связанных файлов.
