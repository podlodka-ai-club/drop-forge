## Context

В текущей конфигурации приложения есть поле `Config.OpenAIAPIKey`, загрузка `OPENAI_API_KEY` из окружения, строка в `.env.example`, упоминание в README и тестовые ожидания. При этом выполнение агентских шагов идет через Codex CLI (`codex exec ...`), а приложение не создает OpenAI-клиент, не валидирует этот ключ и не передает его в runner-ы.

Изменение локальное для конфигурационного слоя и документации. Поток orchestration, Linear, GitManager, proposal/apply/archive runner-ы и формат команд Codex CLI не должны меняться.

## Goals / Non-Goals

**Goals:**

- Удалить `OPENAI_API_KEY` из публичного контракта runtime-конфигурации приложения.
- Оставить Codex CLI ответственным за собственную аутентификацию через свое окружение и настройки.
- Синхронизировать `.env.example`, README и тесты с фактически поддерживаемыми ключами.

**Non-Goals:**

- Не менять способ запуска Codex CLI и набор аргументов `codex exec`.
- Не добавлять новый механизм аутентификации агента.
- Не удалять или менять конфигурацию Linear, GitHub, Logstash, polling interval и runner path settings.
- Не менять OpenSpec orchestration flow и состояния Linear-задач.

## Decisions

1. Удалить ключ из приложения, а не переименовывать его.

   `OPENAI_API_KEY` не используется кодом приложения как runtime-зависимость. Переименование оставило бы тот же ложный контракт и требовало бы поддержки еще одного неиспользуемого параметра.

2. Не передавать `OPENAI_API_KEY` явно в окружение Codex CLI.

   Текущий executor запускает внешний CLI и наследует процессное окружение стандартным способом. Приложение не должно знать провайдерские секреты агента, если само не вызывает OpenAI API.

3. Ограничить тестовые изменения конфигурационным контрактом.

   Нужны проверки, что `Load()` больше не читает и не возвращает `OPENAI_API_KEY`, а `.env.example` больше не содержит этот ключ. Интеграционные тесты orchestration не требуют изменений, потому что их поведение не зависит от ключа.

## Risks / Trade-offs

- Разработчики, которые считали `.env.example` источником настройки Codex CLI, больше не увидят там `OPENAI_API_KEY` -> README должен явно отражать только параметры приложения, чтобы не смешивать настройки внешних инструментов с runtime-конфигом orchestrator-а.
- Если в будущем приложение начнет напрямую вызывать OpenAI API, потребуется отдельное изменение OpenSpec и возврат новой конфигурации с понятным владельцем и тестами -> текущее изменение сознательно фиксирует отсутствие такой зависимости.
- Удаление поля `Config.OpenAIAPIKey` является compile-time breaking change для внутреннего кода, который мог появиться параллельно -> реализация должна выполнить `rg OPENAI_API_KEY|OpenAIAPIKey` и обновить все ссылки.
