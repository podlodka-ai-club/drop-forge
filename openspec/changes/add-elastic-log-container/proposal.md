## Why

Сейчас у проекта нет локального сервиса для долговременного хранения и поиска логов, поэтому сценарии отладки оркестрации остаются привязаны к stdout процесса. Нужен минимальный Docker-based Elastic, чтобы на раннем этапе получить единое место для логов без усложнения архитектуры приложения.

## What Changes

- Добавить локальный Docker Compose сервис Elasticsearch для хранения логов разработки.
- Описать runtime-параметры подключения к Elastic через `.env.example` без секретов и значений по умолчанию.
- Централизованно загрузить параметры Elastic в существующем пакете конфигурации Go.
- Задокументировать минимальный сценарий запуска и проверки доступности контейнера.
- Не добавлять ingestion pipeline, Kibana или сложную схему индексов в рамках этого изменения.

## Capabilities

### New Capabilities

- `elastic-log-storage`: локальная инфраструктура Elasticsearch для хранения логов и параметры подключения приложения к ней.

### Modified Capabilities

- Нет.

## Impact

- Новые или измененные файлы инфраструктуры: `docker-compose.yml`, возможно `.gitignore` для локальных данных Elastic.
- Конфигурация: `.env.example`, `internal/config`.
- Тесты: существующие тесты конфигурации и новые проверки загрузки параметров Elastic.
- Документация: краткое описание запуска локального Elastic и проверки health endpoint.
- Внешняя зависимость: официальный Docker image Elasticsearch без добавления Go-библиотек на этом этапе.
