## Why

Сейчас один orchestration pass обрабатывает proposal, apply и archive задачи последовательно. Долгая задача в одной колонке блокирует запуск задач из других колонок, хотя эти стадии независимы и могут выполняться одновременно.

## What Changes

- Запускать задачи из `Ready to Propose`, `Ready to Code` и `Ready to Archive` в отдельных горутинах по стадиям, чтобы работа в одной колонке не блокировала другие колонки.
- Сохранять текущий lifecycle каждой отдельной задачи: перевод в in-progress до runner-а, запуск соответствующего runner-а, затем перевод в review state после успеха.
- Возвращать и логировать ошибки с контекстом стадии и задачи, не останавливая уже запущенные задачи из других стадий.
- Дождаться завершения всех запущенных goroutine в рамках одного pass перед завершением pass и переходом monitor-а к следующему polling interval.
- Не добавлять внешний queue/scheduler и не менять workflow самих proposal/apply/archive runner-ов.

## Capabilities

### New Capabilities

- Нет.

### Modified Capabilities

- `proposal-orchestration`: изменить orchestration runtime так, чтобы задачи из разных входных колонок могли выполняться параллельно в рамках одного pass.

## Impact

- `internal/coreorch`: планирование задач по стадиям, запуск stage worker-ов в goroutine, сбор ошибок, тесты конкурентного поведения.
- `cmd/orchv3`: wiring остается прежним, но default runtime получит параллельное исполнение внутри существующего monitor-а.
- `architecture.md`: нужно обновить описание текущего маппинга и потоков, потому что меняется взаимодействие стадий внутри `CoreOrch`.
- `openspec/specs/proposal-orchestration/spec.md`: обновить требования о последовательной обработке и добавить требования к параллельному выполнению по стадиям.
