## Context

`CoreOrch` уже получает managed tasks одним вызовом `TaskManager.GetTasks(ctx)` и маршрутизирует их в три stage route: proposal, apply и archive. Сейчас `RunProposalsOnce` проходит по задачам в возвращенном порядке и синхронно вызывает `processProposalTask`, `processApplyTask` или `processArchiveTask`. Из-за этого длительный runner одной стадии задерживает старт задач в других колонках.

Важное ограничение: in-progress колонки уже защищают задачу от повторного выбора следующим polling pass. Поэтому для этого изменения достаточно параллелить независимые stage route внутри одного pass, не вводя distributed lock, lease или отдельный scheduler.

## Goals / Non-Goals

**Goals:**

- Запускать proposal, apply и archive route в отдельных goroutine в рамках одного orchestration pass.
- Позволить задачам из разных ready-колонок выполняться одновременно.
- Сохранить последовательную обработку внутри каждой стадии, чтобы не менять текущие runner-контракты и не создавать конкуренцию за stage-specific ресурсы.
- Дождаться завершения всех stage goroutine перед возвратом из pass.
- Вернуть агрегированную ошибку pass, если одна или несколько stage goroutine завершились ошибкой, и при этом не отменять автоматически другие уже запущенные стадии.

**Non-Goals:**

- Не добавлять worker pool, per-stage concurrency limit или runtime-конфигурацию числа worker-ов.
- Не менять поведение `ProposalRunner`, `ApplyRunner` и `ArchiveRunner`.
- Не менять модель `TaskManager` и набор Linear workflow state IDs.
- Не реализовывать retry/backoff/lease поверх существующих in-progress переходов.
- Не запускать несколько задач одной и той же колонки параллельно.

## Decisions

### D1. Группировать задачи по stage перед запуском goroutine

После `GetTasks` `CoreOrch` должен разложить задачи по трем спискам: proposal, apply, archive. Для каждого непустого списка запускается отдельная goroutine, которая последовательно обрабатывает задачи этой стадии текущими helper-ами.

Альтернатива: запускать goroutine на каждую задачу. Отброшено, потому что это сразу вводит неограниченный параллелизм, усложняет порядок переходов внутри одной колонки и может одновременно запустить несколько heavy Codex/git процессов одной стадии.

### D2. Сохранять stage-local sequential order

Внутри списка одной стадии порядок должен остаться таким же, как порядок задач из `TaskManager.GetTasks`. Это сохраняет предсказуемость текущих тестов и поведения для нескольких задач одной колонки.

Альтернатива: сортировать задачи внутри стадии или обрабатывать их в map order. Отброшено, потому что `TaskManager` уже является источником порядка, а дополнительная сортировка не требуется для цели изменения.

### D3. Собирать ошибки через channel и возвращать агрегированную ошибку

Каждая stage goroutine возвращает не более одной ошибки: первую ошибку, которая остановила обработку этой стадии. `RunProposalsOnce` дожидается всех стадий, собирает ошибки и возвращает агрегированную ошибку с названиями стадий. Успешные стадии при этом завершаются обычным образом.

Альтернатива: отменять общий context при первой ошибке. Отброшено для первого slice, потому что исходная цель - не блокировать другие колонки; автоматическая отмена снова связала бы независимые стадии. Внешняя отмена через переданный `ctx` остается обязательной и должна передаваться во все runner-ы и операции `TaskManager`.

### D4. Общий logger остается shared dependency

Stage goroutine могут писать в один `steplog.Logger`, как и текущий monitor пишет события из одного pass. Если текущая реализация logger-а не гарантирует атомарность записи при параллельном доступе, нужно добавить минимальную синхронизацию на уровне `CoreOrch` или logger writer wrapper, не меняя формат событий.

Альтернатива: отдельный buffer на stage и flush после завершения. Отброшено, потому что это скрывает live-события долгих runner-ов и усложняет отладку.

### D5. `architecture.md` обновляется вместе с реализацией

Изменение нетривиально меняет взаимодействие стадий внутри `CoreOrch`, поэтому маппинг на код и описание текущих потоков нужно обновить: один pass по-прежнему получает задачи один раз, но stage route выполняются параллельно.

## Risks / Trade-offs

- [Больше одновременных внешних процессов] -> Митигировать stage-level лимитом: максимум три параллельные goroutine, по одной на proposal/apply/archive.
- [Concurrent calls to one `TaskManager`] -> Митигировать тем, что каждая задача уже уникальна и переводится по своему task ID; тестами покрыть перекрывающиеся стадии. Если конкретная реализация клиента окажется не thread-safe, добавить минимальную синхронизацию в адаптере или в `CoreOrch`.
- [Interleaved logs] -> Митигировать stage/task identity в сообщениях и атомарной записью JSON Lines.
- [Одна стадия падает, другие продолжают менять задачи] -> Это принятый trade-off ради независимости колонок. Ошибка pass все равно возвращается monitor-у и логируется, а успешные стадии не откатываются.
- [Следующий polling pass стартует слишком рано] -> `RunProposalsOnce` должен ждать все goroutine, поэтому monitor переходит к polling interval только после завершения всех stage route.
