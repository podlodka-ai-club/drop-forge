## Context

`CoreOrch.RunProposalsOnce` сейчас загружает managed tasks один раз и в одном цикле маршрутизирует их по state ID. Proposal, Apply и Archive handlers выполняют полный lifecycle задачи синхронно: move to in-progress, runner, move/attach review. Долгие операции находятся внутри runners, поэтому последовательный цикл блокирует независимую apply-задачу на время proposal-задачи и наоборот.

Требование DRO-34 ограничено proposal и apply. Archive уже включен в общий runtime, но в этой итерации не входит в параллельный scheduling, чтобы не расширять область риска.

## Goals / Non-Goals

**Goals:**

- Разрешить одновременное выполнение одной proposal-задачи и одной apply-задачи в одном orchestration pass.
- Сохранить существующие state transitions, PR attachment и failure semantics внутри каждой отдельной задачи.
- Не добавлять новые runtime-параметры, внешние зависимости или публичные API.
- Сохранить тестируемость через существующие fake `TaskManager`, `ProposalRunner` и `ApplyRunner`.

**Non-Goals:**

- Не запускать несколько proposal-задач параллельно между собой.
- Не запускать несколько apply-задач параллельно между собой.
- Не менять Archive-route и не запускать Archive параллельно с proposal/apply в этой задаче.
- Не вводить полноценный scheduler, worker pool, retry/backoff или persisted locks.

## Decisions

1. Ввести небольшой stage scheduler внутри `internal/coreorch`, а не отдельный пакет.

   Рационал: изменение локально для orchestration loop и не требует новой переиспользуемой абстракции. Альтернатива с отдельным scheduler-пакетом преждевременна: пока есть только два параллельных stage-слота.

2. Группировать задачи по stage и запускать максимум две горутины: одну для proposal-очереди и одну для apply-очереди.

   Внутри каждой горутины задачи своего stage обрабатываются последовательно в порядке, в котором их вернул `TaskManager`. Это выполняет требование "одновременно делать две задачи" без изменения устойчивости runner'ов к нескольким одновременным клонам/branch updates одного stage.

3. Не отменять соседний stage при ошибке одного stage.

   Если proposal падает, уже запущенный apply должен завершить свой lifecycle, и наоборот. После завершения всех запущенных stage-горутин `RunProposalsOnce` возвращает агрегированную ошибку с контекстом каждого failed stage. Это сохраняет текущий monitor contract: loop логирует pass failure и продолжает следующий pass.

4. Оставить Archive последовательным после завершения proposal/apply stage-горутин.

   Archive не входит в DRO-34. Выполнение archive после proposal/apply снижает риск пересечения с apply на одной ветке задачи и сохраняет предсказуемость текущего archive flow.

5. Структурные логи писать из stage handlers как сейчас, но общий writer должен использоваться так, чтобы конкурентные записи не ломали отдельные log events.

   Если текущий logger/writer уже синхронизирует запись полного события, дополнительная защита не нужна. Если нет, `CoreOrch` должен обернуть writer mutex-оберткой на время pass/loop wiring.

## Risks / Trade-offs

- Одновременные записи в fake/test dependencies могут выявить data race в тестовых double'ах -> обновить тестовые helpers с mutex там, где они используются конкурентно.
- Ошибка одного stage больше не останавливает другой stage немедленно -> это намеренный trade-off ради независимого прогресса; mitigation: возвращать агрегированную ошибку после join и логировать task-level failures.
- Максимум две одновременные задачи ограничивает throughput -> это принято как минимальное решение для DRO-34; расширение до configurable concurrency можно добавить отдельной спецификацией после подтверждения необходимости.
- Archive остается последовательным -> это снижает scope и риск branch conflict, но не ускоряет archive-задачи в этом изменении.
