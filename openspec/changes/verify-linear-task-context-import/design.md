## Context

`TaskManager` уже возвращает Linear task payload с `id`, `identifier`, `title`, `description`, `state` и `comments`. `proposal-orchestration` уже отвечает за выбор задач в `ready to propose` и подготовку входной строки для proposal runner. Linear-задача `DRO-28` используется как контрольный пример: заголовок сам по себе не несет смысла, а полезный контекст находится в описании и комментарии.

## Goals / Non-Goals

**Goals:**

- Гарантировать, что proposal runner получает описание Linear-задачи и комментарии в дополнение к идентификатору и заголовку.
- Покрыть это unit-тестом на payload, эквивалентный `DRO-28`.
- Оставить формат входа человекочитаемым и пригодным для Codex/OpenSpec proposal prompt.

**Non-Goals:**

- Не менять Linear client queries, если текущая модель уже передает description/comments в orchestration layer.
- Не добавлять фильтрацию, ранжирование или резюмирование комментариев.
- Не менять workflow states, PR attachment flow или поведение proposal runner после получения input.

## Decisions

1. Проверять поведение на уровне proposal orchestration, а не Linear client.

   Причина: задача проверяет, что описание и комментарии доходят до proposal agent. Linear client уже покрывается `linear-task-manager`; риск потери контекста находится в месте сборки runner input.

   Альтернатива: добавить интеграционный тест Linear API. Это даст меньше пользы для текущего риска и потребует внешней инфраструктуры.

2. Сохранять явные секции `Linear task`, `Description`, `Comments` во входе runner.

   Причина: явные секции проще тестировать и они устойчивее для LLM-потребителя, чем неструктурированный текст.

   Альтернатива: передавать JSON. Это удобнее для строгого парсинга, но текущий proposal runner принимает человекочитаемый task description; смена формата была бы более широкой.

3. Комментарии передавать в исходном порядке, полученном от `TaskManager`.

   Причина: порядок должен оставаться ответственностью Linear-facing слоя или внешнего API; orchestration не должен добавлять скрытую сортировку без отдельной необходимости.

   Альтернатива: сортировать по времени в orchestration. Это добавит ответственность и потенциально изменит поведение при неполных timestamp.

## Risks / Trade-offs

- Формат input остается текстовым, поэтому строгий downstream parsing не гарантируется -> Митигация: использовать стабильные заголовки секций и unit-тесты на включение ключевых полей.
- Все комментарии передаются без фильтрации, включая нерелевантные -> Митигация: это соответствует текущему scope HITL; фильтрацию можно добавить отдельным change при появлении правил.
- Тестовый кейс `DRO-28` содержит минимальный текст -> Митигация: тест проверяет именно путь доставки description/comments, а не качество содержимого proposal.

## Migration Plan

Миграция данных не требуется. Изменение внедряется как локальное уточнение форматирования input и тестов. Откат возможен возвратом к предыдущей функции сборки input без изменения конфигурации или Linear state.
