## Context

Сейчас проект состоит из минимального Go CLI и централизованной загрузки конфигурации из окружения. Интеграции с внешними системами отсутствуют, поэтому Linear-модуль станет первым внешним адаптером и задаст паттерн для будущих модулей.

Linear предоставляет GraphQL API, через которое можно читать issue, менять state и работать с комментариями. В проекте уже есть правило избегать лишних зависимостей и выносить runtime-конфигурацию в `.env`, поэтому дизайн должен опираться на стандартную библиотеку Go и централизованный конфиг.

Параллельно нужно зафиксировать верхнеуровневую документацию модулей. Если этого не сделать сейчас, первые интеграции быстро разъедутся: часть знаний останется в спеках, часть в коде, часть в головах. Для ранней стадии проекта нужен легкий, но обязательный формат.

## Goals / Non-Goals

**Goals:**
- Добавить простой модуль доступа к Linear issue и комментариям.
- Поддержать операции `GetIssue`, `ListIssues`, `MoveIssue`, `ListComments`, `CreateComment`.
- Использовать Linear identifier вида `ENG-123` как единственный внешний идентификатор задач на первом этапе.
- Добавить минимальную CLI-команду для получения списка задач с одной Linear-доски, заданной через ENV, как временную проверочную точку поверх модуля.
- Централизовать настройки Linear в существующем конфиге приложения.
- Зафиксировать формат верхнеуровневой документации модулей и применить его к Linear.
- Сохранить минимальную архитектуру без преждевременных слоев и фабрик.

**Non-Goals:**
- Массовые операции над issue.
- Поддержка webhooks, подписок или двусторонней синхронизации.
- Полное покрытие доменной модели Linear.
- Поиск или обновление задач по internal ID Linear.
- Поддержка rich text представления комментариев.
- Визуализация workflow в UI оркестратора.
- Автоматическое построение документации из кода.

## Decisions

### 1. Модуль Linear оформляется как отдельный пакет-адаптер с узким API

Предлагается добавить пакет уровня `internal/linear` с типом сервиса/клиента, который скрывает детали GraphQL и отдает наружу только нужные проекту структуры:

- `Issue` c полями `ID`, `Identifier`, `Title`, `Description`, `StateName`, `TeamID`
- `Comment` c полями `ID`, `Body`, `AuthorName`, `CreatedAt`
- методы `GetIssue`, `ListIssues`, `MoveIssue`, `ListComments`, `CreateComment`

Почему так:
- проект ранний, поэтому достаточно одного адаптера без выделения дополнительного transport-слоя;
- API узкое и совпадает с реальными сценариями оркестратора;
- детали схемы Linear не просачиваются в остальной код.
- внешний контракт опирается только на identifier задач, чтобы не смешивать пользовательский формат `ENG-123` и internal ID Linear.

Альтернативы:
- использовать универсальный GraphQL-клиент или SDK;
  отклонено, потому что это первая интеграция и стандартного `net/http` достаточно;
- сразу вводить отдельные интерфейсы `Repository`/`Provider`;
  отклонено как преждевременная абстракция до появления второй реализации или тестовой боли.

### 2. Для работы с Linear использовать прямые GraphQL-запросы через стандартную библиотеку Go

Реализация должна использовать `net/http`, `encoding/json` и локальные структуры запросов/ответов. Точка входа API фиксируется в модуле, а из окружения приходит только токен.

Почему так:
- соответствует правилу проекта не тянуть внешние инструменты без необходимости;
- GraphQL-схема нужна только частично, и проще контролировать ровно те поля, которые используются;
- упрощается диагностика ошибок и тестирование через `httptest`.

Альтернативы:
- внешний Linear SDK;
  отклонено из-за лишней зависимости и слабой необходимости на текущем этапе;
- generic GraphQL client;
  отклонено, потому что для четырех операций проще явные запросы.

### 3. Смена столбца реализуется как смена workflow state по имени или ID

Публичная операция `MoveIssue` должна принимать identifier issue и целевой state reference. Если передан ID state, модуль использует его напрямую. Если передано имя, модуль сначала читает доступные состояния команды issue и резолвит нужный state перед обновлением.

Почему так:
- пользователь мыслит “столбцами”, а в Linear это workflow state;
- на практике имя состояния удобнее в сценариях оркестратора, но ID нужен для детерминизма и автоматизации;
- резолвинг внутри модуля не заставляет верхний слой знать детали workflow.

Альтернативы:
- поддерживать только state ID;
  отклонено как неудобный API для человека и агентов;
- поддерживать только имя state;
  отклонено, потому что ID полезен для стабильных автоматизаций.

### 4. Комментарии возвращаются плоским списком в хронологическом порядке

На первом этапе модуль читает комментарии issue как плоский список без реконструкции тредов, реакций, вложенных сущностей и rich text. Поле `Body` трактуется как обычный текст, пригодный для логов, CLI и внутреннего use case. Для записи комментария модуль валидирует непустое тело до отправки запроса.

Почему так:
- это покрывает базовую коммуникацию агента с задачей;
- не раздувает модель данных;
- упрощает тесты и интеграцию с будущим оркестратором.

Альтернативы:
- сразу поддерживать иерархию replies;
  отклонено до появления явного кейса.
- сразу нормализовать rich text в markdown/plain text;
  отклонено, потому что на первом этапе достаточно обычного тела комментария без форматирования.

### 5. Первый пользователь модуля — внутренний use case, CLI остается временной проверочной точкой

Основной потребитель Linear-модуля будет жить во внутреннем use case оркестратора. Чтобы модуль можно было проверить вручную до появления полного сценария оркестрации, нужно добавить минимальную CLI-команду для получения списка задач из Linear.

CLI-команда не должна становиться отдельным слоем бизнес-логики: она только загружает конфиг, создает Linear-клиент, вызывает `ListIssues` для одной доски, заданной ENV-переменной, и печатает identifier, state и title. Для первого этапа доска задается через `LINEAR_BOARD_ID`; если переменная не задана, CLI-команда должна завершаться ошибкой конфигурации.

Почему так:
- внутренний use case остается целевой архитектурной точкой;
- CLI помогает быстро проверять токен, доступ к API и форму ответа;
- логика не дублируется между CLI и модулем.
- список задач не размывается по всему Linear workspace и остается привязан к рабочей доске команды.

Альтернативы:
- сразу реализовать только внутренний use case без CLI;
  отклонено, потому что пока нет полного сценария использования, а ручная проверка интеграции нужна;
- сделать полноценный CLI для всех операций Linear;
  отклонено как преждевременное расширение публичного интерфейса.
- выводить задачи текущего пользователя или всей команды без явной доски;
  отклонено, потому что нужен контролируемый список задач с конкретной доски.

### 6. Верхнеуровневая документация модулей хранится в `docs/modules`

Для каждого модуля создается отдельный документ в `docs/modules/<module>.md`, а в `docs/modules/README.md` поддерживается индекс. Документ модуля должен содержать единый набор разделов:

- назначение;
- границы ответственности;
- публичные операции;
- зависимости и внешние системы;
- конфигурация;
- отказные сценарии и ограничения;
- связанные спеки/изменения.

Почему так:
- документация остается на уровне модуля, а не распадается по пакетам;
- путь легко discoverable и не требует специальной генерации;
- подход достаточно легкий для ранней стадии, но уже создает дисциплину.

Альтернативы:
- держать документацию только в `README.md` корня;
  отклонено, потому что модульные детали быстро смешаются;
- класть `README.md` внутрь каждого пакета;
  отклонено, потому что один модуль может состоять из нескольких пакетов, а нужна общая верхнеуровневая картина.

## Risks / Trade-offs

- [Риск] В Linear разные команды могут иметь одинаковые названия состояний → [Смягчение] резолвить state только в пределах команды конкретной issue и отдавать явную ошибку при неоднозначности.
- [Риск] GraphQL-ответы могут содержать частичные ошибки и данные одновременно → [Смягчение] ввести единый разбор ошибок ответа и оборачивать ошибки контекстом операции.
- [Риск] Первая интеграция может задать слишком специфичный паттерн → [Смягчение] держать API модуля минимальным и не выделять абстракции без повторного использования.
- [Риск] Документация модулей начнет отставать от кода → [Смягчение] включить обновление `docs/modules` в задачи change и требовать синхронного обновления при изменении публичного поведения или конфига.

## Migration Plan

1. Добавить переменные токена Linear и `LINEAR_BOARD_ID` в конфиг и `.env.example`.
2. Реализовать пакет `internal/linear` с HTTP-клиентом и GraphQL-запросами.
3. Покрыть публичные операции модуля table-driven и `httptest`-тестами.
4. Добавить минимальную CLI-команду для получения списка задач с Linear-доски из `LINEAR_BOARD_ID`.
5. Добавить `docs/modules/README.md` и документ `docs/modules/linear.md`.
6. Позже подключить модуль к внутреннему use case оркестратора, когда появится полный сценарий.

Откат простой: удалить вызовы нового модуля и переменную конфигурации, так как изменение не требует миграции данных.
